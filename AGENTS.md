## 1. 文档定位与优先级
- 本文件是本仓库的协作与进度单一事实源（Single Source of Truth）。
- 用于维护两类信息：
  - 开发规范与安全红线（可执行、可验证）。
  - 开发进度看板（里程碑与任务级状态）。
- 优先级规则：若与全局/会话规则冲突，以“更严格、更安全、更可验证”的规则为准。

## 2. 协作规范

### 2.1 语言与沟通
- 默认中文，明确要求英文时再切换。
- 所有“完成”结论必须附带命令输出或可复现证据。

### 2.2 实施规范
- 先读后改：先确认边界、影响范围、回滚点，再执行变更。
- 复用优先：优先使用既有函数/组件，避免复制粘贴同段逻辑。
- 边界校验：对外接口与边界输入必须做空值、类型、范围、格式校验。
- 性能底线：避免明显 N+1、无界循环、全量加载；不可避免时说明上限。
- 错误处理：返回/抛错策略一致，日志不得泄露敏感信息。

### 2.3 安全红线
- 禁止提交、打印、回显敏感信息（key、token、cookie、私钥、连接串、.env 内容）。
- 禁止执行来源不明脚本、破坏性命令、未审计下载命令。
- 涉及用户数据/权限/支付时，先写威胁模型（1-3 条）再改代码。

### 2.4 Git 纪律
- 单个提交只做一件事，禁止功能改动与无关重构混杂。
- Commit message 使用 Conventional Commits：
  - `feat|fix|chore|docs|refactor|test|build|ci`
- 提交说明必须包含：
  - What
  - Why
  - How to verify（命令 + 期望结果）

## 3. 质量门禁（当前仓库）
- 当前可执行门禁命令：
  - `cd go-api && go test ./...`
  - `cd py-rag-service && python -m pytest -q`
  - `cd py-worker && python -m pytest -q`
  - `docker compose config`
- 严禁“无证据标记 done”：任务状态改为 `done` 前，必须填写验证命令与证据摘要。

## 5. 维护机制（强制）
1. 开始任务前：将任务状态置为 `in_progress`。
2. 完成任务后：先填验证命令与证据，再置为 `done`。
3. 遇阻塞：立刻置为 `blocked`，并在阻塞记录表补齐“原因/影响/解除条件”。
4. 每次提交必须同步更新 AGENTS 看板。
