# 产品定义

## 产品名称

RAG-QA System

## 一句话定义

一个面向私有知识库场景的检索增强问答系统，重点解决“文档上传后状态不可见、问答结果缺少依据、运维排障成本高”三个常见问题。

## 目标用户

### 直接使用者

- 需要基于内部文档发起问答的业务成员
- 需要维护知识库内容的管理员
- 需要查看上传、索引、失败原因的运维或研发人员

### 采购或评估者

- 想验证 RAG 方案是否具备工程可落地性的团队负责人
- 需要演示知识库上传、入库、检索、问答全链路的售前或交付人员

## 产品要解决的问题

### 1. 文档入库是黑盒

许多 RAG Demo 只展示“上传文件”与“提问”两个动作，中间发生了什么不可见。一旦解析失败、向量库不可达或任务堆积，用户只能看到页面一直转圈。

### 2. 问答结果难以追责

如果回答没有引用信息，团队很难判断答案是否真来自知识库，还是模型根据常识补全甚至幻觉。

### 3. 运维成本被低估

真实系统不是只有模型调用。它还涉及对象存储、数据库、向量库、消息队列、文件解析、重试与失败恢复。

## 核心价值

- 把知识库入库过程显式化，而不是隐藏在后台任务里
- 把问答请求结构化，而不是只返回一段文本
- 把排障入口前置到产品界面与脚本，而不是让维护者从零开始翻容器日志

## 当前能力范围

### 知识库管理

- 创建知识库
- 删除单个知识库或批量删除知识库
- 查看知识库下的文档列表

### 文档管理

- 上传 `txt / pdf / docx`
- 查看文档详情
- 查看文档处理时间线
- 在线预览文档
- 对小型 TXT 文档执行在线修改
- 删除文档及其关联向量数据

### 入库可观测性

- 查询入库任务状态与进度
- 记录入库阶段事件
- 在前端展示当前阶段、最近更新时间、任务 ID、文档 ID
- 提供日志排障命令

### 问答能力

- 登录后发起对话
- 单知识库或多知识库作用域问答
- 普通请求与流式请求
- 会话列表与消息历史
- 消息反馈

## 角色边界

| 角色 | 当前能力 |
| --- | --- |
| Admin | 登录、聊天、知识库管理、文档管理、评测页访问 |
| Member | 登录、聊天 |

当前系统未提供注册、组织、租户、细粒度权限模型。

## 非目标

以下内容不在当前仓库的目标范围内：

- 完整的企业级 IAM / SSO 集成
- 多租户隔离和计费体系
- 高可用集群部署模板
- 审计合规体系
- 复杂工作流编排平台

## 适用场景

- 技术文档、流程文档、产品资料的内部问答
- 演示“RAG 不只是大模型接口，而是一套完整工程链路”
- 作为团队内部知识库平台的 MVP 或二次开发底座

## 不适用场景

- 直接用于公网多租户 SaaS 商业化环境
- 需要零信任安全体系与细粒度合规审计的行业
- 需要原生支持海量组织、多部门隔离和复杂审批流的系统

## 成功标准

如果一个首次接触仓库的人能够做到以下事情，说明产品说明达标：

1. 在本地启动全部服务并成功登录
2. 上传文档并能看到入库阶段变化
3. 基于文档完成一次问答
4. 能在上传卡住时通过日志和运行手册定位问题

## 后续演进方向

- 更细的权限模型和审计日志
- 更稳定的流式响应观测与指标上报
- 更完整的评测数据、基线报告和回归报告
- 更明确的部署模板与公开发布资产
