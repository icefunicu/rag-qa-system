# RAG-QA AI 应用工程师面试主审问答稿

> 使用方式
>
> - 这份文档不是项目总结，而是一份“能拿去面试”的主审问答稿。
> - 它的目标不是让你背概念，而是让你把项目讲到：流程清楚、做法具体、参数有数、trade-off 明白、边界守得住。
> - 内容严格基于已经确认过的实现。已经做过的直接说，没明确做过的只会当优化思路讲。

## 一、先把项目讲透：你到底做了什么

### 问题 1：你先别讲技术栈，直接说这个项目是干什么的

最稳的回答不是“这是一个 RAG 系统”，而是：

这是一个工程化的企业知识问答系统，它把知识接入、知识治理、检索编排、答案生成、稳定性控制和评测回归接成了一条完整链路。也就是说，它关注的不只是“模型能不能回答”，还关注“知识怎么进来、能不能查准、回答有没有证据、系统稳不稳、改完之后怎么验证没退化”。

如果面试官继续问“那它和普通 demo 的区别是什么”，可以接着说：

普通 demo 往往只做三步：上传文档、向量检索、把结果拼给模型。这个项目多做了几层真正工程上重要的事情：

1. 多源接入，不是只有上传 PDF。
2. 入库 worker 化，不是同步接口跑全链。
3. 检索不是单一路径，而是结构、全文、向量三种信号一起工作。
4. 回答前先判断证据够不够，再决定是否能答。
5. 有会话预算、幂等、限流、降级、回归评测这些配套能力。

所以我会把它讲成“工程化 RAG 平台”，而不是“一个能跑的问答 demo”。

### 问题 2：你用 1 到 2 分钟把整套主链路讲一遍

可以这样回答：

整套链路分成前后两段。前面是知识入库，后面是在线问答。

知识入库这边，系统支持多种知识源，包括文件上传、本地目录、Notion、网页和 SQL 结果。不同来源进来之后，不是分别走完全不同的后处理，而是会先被统一成待处理文档，再由异步 worker 做后续工作。worker 的核心职责是解析内容、切片、保留元数据、做 OCR 增强，再把结果写进结构化存储和 Qdrant。这样一来，后面检索到的不是一堆匿名文本，而是带文档、章节、页码、字符范围和来源类型的证据单元。

在线问答这边，用户问题先进网关，网关负责做会话处理、预算检查、历史读取和知识范围解析，然后把问题 fanout 到一个或多个知识库。每个知识库内部不是只做向量检索，而是先做 query rewrite，再同时跑结构检索、全文检索和语义检索，之后用融合和重排选出最合适的证据。拿到证据以后，系统不会默认模型一定可以回答，而是先判断这次是不是证据足够、风险可控的 grounded answer；如果证据不够，就降级成 common knowledge；风险更高时会直接拒答。最后回答里还会尽量保留引用标记和来源痕迹，方便追溯。

如果要一句话收尾，我会说：

这套系统真正做的不是“让模型更会答”，而是“让系统知道什么时候可以答、凭什么答、答错了怎么追、改了之后怎么验证”。

## 二、知识入库链：从原始知识到可检索证据

### 问题 3：知识到底是怎么进系统的？

这个问题看起来简单，实际上很能暴露系统是不是只做了个上传页面。

当前系统明确支持的知识源至少包括：

1. 普通文件上传
2. 本地目录同步
3. Notion 页面同步
4. 网页内容同步
5. SQL 查询结果同步

为什么这个点值得讲？

因为企业知识几乎从来都不是集中在一个地方。它可能一部分在文档，一部分在知识平台，一部分在网页，一部分干脆在数据库查询结果里。如果系统只支持单一文件上传，那它离真实使用场景还很远。

但更重要的不是“来源多”，而是“来源多以后有没有统一标准”。这个项目的做法是，把这些来源尽量统一成后续可处理的文档对象，再由后面的解析、切片和索引链去接。这样做的意义是，检索层和回答层不需要感知每一种来源的差异。

### 问题 4：为什么入库不用同步接口做完？

因为真正的入库不是保存文件，而是一串长动作：

1. 拿到原始内容
2. 判断是不是新版本
3. 解析正文
4. 做 section / chunk
5. 写结构化存储
6. 做 OCR 或视觉增强
7. 写向量索引
8. 更新状态

把这些全塞进同步接口里，会有几个问题：

1. 请求耗时很长，大文档体验会很差。
2. 中间任一步失败，接口就会显得非常脆弱。
3. 很难做恢复和重试。
4. 用户必须等“全部处理完成”才能用。

所以当前系统用的是异步 worker 模式。这个设计的核心好处不是“更快”，而是“更稳、更好恢复、更适合长链路”。

当前这条 worker 链还有一个很值得讲的设计，就是文档状态不是一次到位，而是分阶段推进。也就是说，系统允许文档先在一个较早阶段就进入可查询状态，后面的 OCR 或视觉增强再继续补齐。这背后的判断很工程化：对很多场景来说，“先有基础可用”比“等所有增强结束再开放”更实用。

### 问题 5：文本解析阶段到底做了什么？

这一步本质上是在做“从原始文件到结构化文本”的转换。

不同文件格式的解析方式不一样，但目标是统一的：

1. 尽量抽出正文
2. 尽量识别章节结构
3. 为后面的切片保留层级信息

你在面试里可以这样讲：

系统不是只把文件读成一整段字符串，而是尽量把它拆成有结构的 section。因为后面无论是切片、检索、引用还是回答，都更依赖“这段内容属于哪一章、哪一节、哪一页”，而不是只依赖纯文本本身。

这一步的真实边界也要敢说：

它支持常见文档格式，但没有做成特别复杂的文档理解框架。它更像一个实用型解析层，而不是文档理解研究项目。

### 问题 6：文本切片具体怎么做？为什么这样做？

这是最容易被深问的地方，必须讲透。

当前系统采用的是字符级滑窗切片，不是 token-based，也不是 semantic chunking。可以抓住四个关键信息来讲：

1. 切片是在 section 的基础上做的，不是从整篇文档直接乱切。
2. 主要窗口大小大约是 `1000` 个字符。
3. overlap 大约是 `100` 到 `120` 个字符。
4. 每个 chunk 都会带着自己的元数据继续往后走。

为什么用这种方案？

因为它是一个很典型的工程折中。

如果 chunk 太小，比如只有两三百字符，精细是精细了，但很容易把一个完整概念切碎。结果就是检索虽然能命中一些字词，但拿到的片段本身不够解释问题。  
如果 chunk 太大，比如上千接近两千甚至更多字符，虽然上下文更完整，但会把很多弱相关内容一起塞进去，导致 embedding 和后续 prompt 都更容易被噪声污染。

所以 `1000` 左右的字符窗口，本质是在“完整性”和“噪声/成本”之间取一个中间值。

那 overlap 为什么需要？

因为 chunk 边界是最容易丢语义的地方。尤其是定义句、长句、跨句关系、表述承接，如果完全不重叠，边界附近的信息很容易被截断。适当 overlap 的作用，就是让边界信息能在两个 chunk 中都出现一次，降低被切断的风险。

这里我会主动说两个局限，显得更专业：

1. 这是字符切片，不是 token 切片，所以它和模型实际 token 消耗不是一一对应。
2. 它不理解真正的语义边界，所以在段落结构复杂时，仍然可能把一个完整语义拆开。

如果面试官追问“那为什么你们不直接做 semantic chunking”，最稳的回答是：

因为当前阶段团队优先把多源接入、索引、检索和回答主链做实了。字符滑窗的优点是稳定、透明、容易统一处理不同来源文档。语义切片当然值得做，但它属于下一层优化，不是当前系统的主卖点。

### 问题 7：切片之后为什么还要保留那么多 metadata？

这是很多人答不出来，但一答出来就很加分的问题。

当前系统保留的元数据不是装饰，而是后面多处逻辑都依赖它。你可以从四个用途讲：

1. 检索过滤  
   系统要知道这段内容属于哪个知识库、哪个文档、哪一类来源，才能做范围控制。

2. 排序与重排  
   正文、OCR、视觉区域这几类内容的可靠性不完全一样，后面排序和重排如果完全不知道来源，就很难做好。

3. prompt 组织  
   模型不是只看到文本，还能看到这段证据来自哪里、在哪一页、属于哪个章节，这样更容易形成 grounded answer。

4. 引用返回  
   最终如果要告诉用户“这句话来自哪里”，前面就必须把来源和位置一路带下来。

所以我通常会用一句话总结：

这个系统不是把 chunk 当成匿名文本，而是把它当成带出处的证据单元。

### 问题 8：OCR 和视觉增强在这个系统里到底算什么？

这里一定要讲得准确，不要讲成“我们做了端到端多模态”。

更稳的回答是：

它是一种文档理解增强能力。系统会把 PDF、DOCX 或图片里的视觉内容抽出来，再通过 OCR 变成文字和区域信息，然后并进现有的文本检索链路。这样一来，扫描件、截图或者图文混排页面里的信息，不会在入库时直接丢掉。

为什么这个点重要？

因为很多企业文档不是干净的数字文本。如果你只处理正文，很多真实知识根本进不到索引里。

但边界也要讲清楚：

这套方案本质上还是“视觉转文本再检索”，不是完整的多模态生成方案。它的收益是扩大可覆盖知识范围，代价是 OCR 会带来噪声，而且表格和复杂版面并没有被完整恢复。

### 问题 9：embedding 和向量索引是怎么做的？

这个问题不能只说“用了 Qdrant 和 embedding 模型”，要讲出为什么这样组合。

当前系统用的是 dense+sparse 混合方案。你可以这么理解：

1. dense embedding 负责语义相似
2. sparse 表示负责词项和关键词信号
3. 两者一起进入同一套检索链

当前比较明确的参数是：

1. dense 模型用的是 `BAAI/bge-small-zh-v1.5`
2. sparse 模型用的是 `Qdrant/bm25`
3. collection 名称是 `kb-evidence`

为什么这种组合合理？

因为企业知识问答里，很多问题既需要“意思相近就能找到”，也需要“术语、字段、制度名要命中得准”。如果只靠 dense，容易出现“意思像，但不够准”；如果只靠 sparse，又会漏掉换一种表达方式的问题。把 dense 和 sparse 组合起来，本质是在补两个世界的短板。

这里也可以顺手讲一个边界：

这个项目里，embedding 的主价值在于“它和后面的混合检索是成套设计的”，而不是“模型本身多先进”。如果面试官一直追问模型选型，你可以稳稳地把话题拉回系统设计，而不是硬辩“这个模型一定最好”。

## 三、在线问答链：从用户问题到最终回答

### 问题 10：用户提问以后，系统第一步做什么？

第一步不是检索，也不是调模型。

更真实的回答是：

系统先把这个问题放回到一个受控的会话上下文里。具体会做几件事：

1. 看这次请求是不是重复请求，避免同一个问题被重复执行。
2. 看当前用户和系统整体并发有没有超限，避免请求把服务打爆。
3. 看当前会话的成本预算是不是已经接近上限，避免多轮对话无限消耗。
4. 读取最近几轮历史消息，准备做轻量上下文化。
5. 解析这次问题的 scope，也就是到底要查哪些知识库、哪些文档范围。

为什么这一步值得讲？

因为它体现的是 AI 应用工程而不是单次模型调用。真实系统里，用户请求不是孤立的一句话，而是有会话、有预算、有权限、有范围的。你如果一上来就只讲“检索和生成”，面试官很容易判断你只看过算法，不懂系统。

### 问题 11：为什么要把 gateway 和 knowledge-base 拆成两层？

这个问题很适合体现架构思路。

可以这样回答：

我理解这里的拆分是有明确职责边界的。knowledge-base 负责知识内容本身的生命周期，包括入库、切片、索引和单库检索；gateway 负责用户请求、会话、范围、跨库聚合和答案生成。也就是说，下层服务更像“知识能力层”，上层服务更像“用户交互和编排层”。

这样拆分有几个实际好处：

1. 单库能力可以独立演进。
2. 多个知识库可以独立治理，不需要一上来就全塞进一个大索引。
3. gateway 可以统一做跨库聚合、预算控制、回答边界控制。
4. 检索和生成耦合度更低，问题定位更清楚。

代价也要讲：

1. 多了一次网络 hop。
2. 下游失败点更多。
3. 聚合层会带来额外时延和规则复杂度。

所以这个项目的一个很明显判断是：它宁愿接受多层服务的复杂度，也不把所有逻辑糊在一个大服务里。

### 问题 12：检索为什么不是简单查一个向量库？

这是整套系统最值得深讲的问题之一。

当前系统的思路不是“把 query 编成向量然后 top-k”，而是把检索拆成多个信号层。

先说流程：

1. query 先做 rewrite
2. 再同时跑结构检索、全文检索、向量检索
3. 三路结果先融合
4. 再做 rerank
5. 再按上下文预算裁剪成最终证据

为什么这么设计？

因为企业知识问答里的 query 类型非常混杂。

有些问题本身就带章节感，比如“发票流程第三步是什么”“安全制度的例外条款怎么规定”；这种问题结构信号很强。  
有些问题是术语或参数型的，比如“某字段的默认值是什么”；这种问题全文和词项命中更稳。  
还有些问题是同义表达、口语化描述，这种时候向量检索才最有价值。

所以这里的核心思路不是“找一种最强检索器”，而是“让不同检索器各自负责自己擅长的部分”。

### 问题 13：query rewrite 的价值到底是什么？

很多人会把 query rewrite 讲得很虚，比如“让查询更优”。更好的讲法是：

它的作用是把用户问题从“自然问法”转成“更适合搜索系统工作的表达”。当前这条链会做几类轻量处理：

1. 去掉疑问词和口语成分
2. 提取问题 focus
3. 做词项扩展
4. 给全文检索和结构检索提供更容易命中的关键词

这背后的判断是：

原问题更适合给模型理解，不一定最适合给检索器工作。检索器需要的往往是更集中、更明确、更偏关键词和结构的信息。

如果面试官追问“为什么不直接用原问题”，可以答：

因为原问题里的自然语言成分太多，尤其是口语化、疑问式表达，会干扰词项检索和结构命中。rewrite 的目标不是美化句子，而是让后面的搜索信号更稳定。

### 问题 14：三路检索之后，为什么还要融合？

因为三类检索器给出的排序语义不一样。

结构检索、全文检索、向量检索各自返回的“分数”不能简单直接比较，所以系统先用加权融合把它们拉到一个统一的候选排序空间。

当前比较关键的参数可以直接记住：

1. 结构信号权重大约是 `1.3`
2. 全文信号权重大约是 `1.0`
3. 向量信号权重大约是 `0.9`

这组参数怎么解释？

最稳的说法是：

当前实现更信任结构命中和全文命中，其次才是向量。不是因为向量不重要，而是因为在当前知识形态下，标题、章节和术语本身就已经是很强的相关性锚点。

如果面试官问“那为什么不是向量最高”，可以答：

在企业知识场景里，很多问题本来就对结构和词项高度敏感。把结构和全文权重放高一点，是为了避免语义检索把“意思差不多但不够准确”的内容推得太前。

### 问题 15：为什么向量候选一开始要取到 40？

这个问题如果答得好，会显得你真的理解检索链。

可以这样讲：

`40` 不是最终返回给用户的结果数，而是向量侧先给后面的融合和重排留一个候选池。因为如果一开始就截得很小，向量检索的长尾价值就很难释放出来；但如果截得太大，又会把后面的排序、上下文和 token 成本一起抬高。

所以这不是一个“算法最优值”，而是一个“候选池上限”的工程参数。它的作用是保证召回有空间，同时不给后续链路造成太大负担。

### 问题 16：为什么融合之后还需要 rerank？

因为融合和 rerank 不是一回事。

融合解决的是“不同检索器的结果怎么合并”。  
rerank 解决的是“合并之后，哪几条最适合作为最终证据”。

你可以用很口语的方式解释：

融合像是把三条不同来源的候选名单并到一起，rerank 像是在并完之后再做一轮精排。没有融合，三路结果不在一个空间里；没有 rerank，融合后的前几条不一定真的是最适合回答问题的证据。

这里还要主动讲一个边界：

当前系统默认的重排更偏 heuristic，也支持更强的外部重排服务。所以在面试里最稳的表述是“项目确实把 rerank 接进了主链，但更强的 cross-encoder 是否在所有环境里都是默认主路径，要看部署配置”。

### 问题 17：gateway 为什么还要再做一轮裁剪？

因为单个知识库内的 top 结果，不等于跨知识库全局最适合给模型看的结果。

当前系统在 gateway 层有比较明确的上下文预算：

1. 每个知识库最多保留 `3` 条
2. 每个文档最多保留 `2` 条
3. 整体最多保留 `8` 条

这三个数字非常值得记，因为它们能体现系统在做什么判断：

它不是追求召回越多越好，而是承认“最后进入模型上下文的证据一定是有限的”，所以必须在更高一层控制证据规模。

这个设计的优点：

1. 简单、清晰、稳定
2. 对 token 成本友好
3. 方便解释和调试

缺点：

1. 比较规则化
2. 不是 query-aware 的动态预算
3. 对特别复杂的问题，未必总能保留最优组合

更高级但不夸张的说法是：

当前实现已经有很明确的上下文预算意识，但还没有演进到更智能的上下文管理框架。

### 问题 18：prompt 是怎么组织的？

不要说“我们有一个很强的 prompt”，那会显得很空。

更成熟的讲法是：

这个系统首先不是所有请求走同一个 prompt，而是至少区分两类模式：

1. grounded answer  
   用在证据足够、可以明确基于知识库回答的场景。

2. common knowledge  
   用在证据不足但问题风险较低的场景，本质上是一种显式降级。

然后，系统不会把检索结果原样粗暴拼接，而是会整理成结构化证据块。证据块里通常会包含：

1. 来自哪个知识库
2. 来自哪个文档
3. 来自哪个章节
4. 页码或字符范围
5. 当前证据分数
6. 引用片段
7. 适量原文

这样做的价值是：

模型看到的不是一堆松散文本，而是一组“有来源、有位置、有优先级”的证据。

### 问题 19：系统怎么决定这次该 grounded 回答、降级回答还是拒答？

这是这套系统很有工程感的一点。

当前逻辑不是“有问题就答”，而是先做证据分级。大致可以这么理解：

1. 证据足够强，走 grounded answer
2. 证据不够，但问题风险低，走 common knowledge
3. 风险高或证据太弱，直接 refusal

这个设计很适合这样解释：

在真实企业场景里，最危险的不是“答不出来”，而是“看起来很自信地答错”。所以系统宁愿在边界不清楚的时候保守一点，也不默认让模型补全。

### 问题 20：怎么降低 hallucination？

这个问题千万别只回答“prompt 写得严格”。

更完整的回答应该分成四层：

1. 先把证据找准  
   用混合检索、融合和重排尽量提升证据质量。

2. 不让弱证据默认进入强回答  
   用证据分级控制 grounded / weak / common knowledge / refusal 路径。

3. 做 prompt 风险控制  
   系统会检查提示注入、试图绕过引用、试图拿到系统 prompt 等风险。

4. 尽量让回答可追溯  
   回答里会尽量保留引用标记和来源信息。

如果面试官问“这样就能彻底防住幻觉吗”，最稳的答法是：

不能，我不会这么夸。它能显著降低“证据不足还硬答”的风险，但还不是最终答案级的 verifier 体系。

### 问题 21：回答生成阶段有哪些关键参数？这些参数怎么理解？

只讲最关键的几个。

1. grounded 回答温度大约是 `0.2`
2. grounded 回答最大输出大约限制在 `1200` 以内
3. common knowledge 回答温度大约是 `0.4`
4. common knowledge 的历史保留更严格，大约只保留少量条数和字符数

这些参数背后的意义其实很清楚：

1. grounded answer 的目标是稳，不是发散，所以温度低。
2. grounded answer 不希望写得过长，否则容易削弱证据和引用的约束力。
3. common knowledge 本来就不是强证据回答，所以会稍微放开一点。

如果面试官问“为什么不是更高温度”，就直接说：

因为这不是创作任务，而是证据约束型问答，稳定比发散更重要。

### 问题 22：短追问是怎么处理的？

这是一个很容易体现“你真的看懂细节”的问题。

当前系统对很短的追问会做轻量级上下文化。简单说就是：

如果用户当前的问题特别短，像“那这个呢”“为什么会这样”“然后呢”这种，系统不会把它当成完全独立的问题，而是会把上一轮用户问题一起带进去理解。

这个做法的优点是：

1. 简单
2. 成本低
3. 对典型 follow-up 场景很实用

它的局限也要讲清楚：

1. 只能覆盖一部分短追问
2. 不是完整的多轮对话推理

所以我会把它讲成“轻量会话上下文化能力”，而不是“完整对话理解系统”。

### 问题 23：为什么回答还要带引用？

因为在企业知识问答里，“可追溯”跟“答得像”一样重要。

引用至少有三层价值：

1. 用户能回到原文看依据
2. 系统调试时知道哪条证据被拿来回答了
3. 后续评测时不仅能看答案对不对，还能看引用是不是合理

但这里也要主动守边界：

当前系统更像在做“引用标记和来源保留”，不是做最终答案逐句逐段的严格事实核验。这一点一定要说清楚。

## 四、性能、成本、稳定性：把系统做成“能长期跑”

### 问题 24：这个项目怎么做性能优化？

这里不要回答“做了很多优化”，要讲清楚优化点在哪里。

当前系统的性能优化更偏“边界控制”，而不是复杂缓存：

1. 入库放到异步 worker，不让长任务占住接口线程
2. section 和 chunk 插入做 batch，降低数据库写入成本
3. 向量检索先控制候选池大小，不让后续排序无限放大
4. gateway 层限制跨库 fanout 并发，默认大约是 `4`
5. 聚合后限制证据数量，避免 prompt 爆炸
6. grounded 回答限制生成长度，大约在 `1200` 以内

这套策略的本质是：

先把最容易失控的地方卡住，再谈更细的优化。

它的优点是简单、稳定、容易解释。  
它的缺点是系统上限不会特别高，尤其在高重复 query 场景里，没有 query cache 会比较可惜。

### 问题 25：这个项目有缓存吗？

这个问题要非常谨慎。

更真实的说法是：

有基础设施级的复用，比如向量库客户端、embedding 相关对象会做复用；但如果你问的是“有没有成熟的 query result cache 或 retrieval cache”，那从当前主链路看并不是系统强项。

所以最稳的回答是：

这个项目当前更擅长通过控制候选规模、控制 fanout、控制上下文长度和控制会话预算来稳住性能和成本，而不是靠复杂缓存架构来提效。

### 问题 26：怎么控制成本？

可以从三层讲：

#### 第一层：检索成本

不是让所有知识库、所有候选都无边界参与。fanout 有并发上限，单库检索有结果上限，聚合后还有全局裁剪。

#### 第二层：上下文成本

不是检索到多少就塞多少，而是明确控制最终证据数量和历史长度，防止 prompt 越堆越大。

#### 第三层：会话成本

系统有会话预算概念，意味着它不是只看一次回答，而是关注整个 session 的累计消耗。

如果面试官问“为什么会话预算重要”，可以答：

因为多轮对话很容易在不知不觉中把历史、证据和输出全部拉长。单次调用看起来不贵，但会话级成本可能失控。

### 问题 27：这个项目怎么做稳定性保障？

这部分非常适合拉开差距。

可以分成四层来讲：

#### 第一层：幂等

避免用户重复提交、网络重试或者前端重复发送导致同一个问题被执行多次。

#### 第二层：并发限制

系统对全局和单用户请求都有上限，不允许某个用户或一波流量把整条问答链拖垮。当前比较明确的上限是全局大约 `32`，单用户大约 `4`。

#### 第三层：worker 重试和死信

入库 worker 不是失败就结束，而是有明确的退避重试。当前能确认的重试节奏大约是：

`5 -> 15 -> 45 -> 135 -> 300` 秒

如果还失败，就会进入死信处理，而不是无限重试。

#### 第四层：下游失败降级

检索层接受部分知识库失败但整体继续；模型层也支持在特定失败场景下切换到备用 route。也就是说，系统不是追求“永不失败”，而是在做“局部失败时整条链别一起倒”。

### 问题 28：模型调用为什么不直接让 SDK 自动重试？

这个问题答好了很显专业。

当前系统的做法更偏“显式 fallback”，而不是“把重试藏在 SDK 里”。也就是说，它更愿意明确控制：

1. 哪些错误值得切换路由
2. 什么时候应该直接暴露问题
3. 哪些失败可以降级

这种设计的优点是行为清楚、定位容易。  
缺点是如果你把 fallback 条件设得很保守，网络抖动时就更依赖外围稳定性设计。

所以最成熟的说法是：

这个系统在模型层更强调“可控的路由和降级”，而不是“尽可能把错误藏起来”。

## 五、评测、回归和工程验证

### 问题 29：你怎么证明这套系统不是“感觉上有效”？

这是一个很关键的问题。

最好的回答不是虚构线上提升，而是说：

这个项目已经把评测和回归做进了工程流程。它有长链路评测脚本，也有 retrieval ablation，对 query rewrite、融合和 rerank 这些模块可以做分组对比；另外还有 regression gate，意味着改动之后不是凭肉眼看几个 case，而是有明确门槛控制。

这套设计的价值不在于“我能马上报出多少提升”，而在于：

1. 复杂链路的每一层都能被验证
2. 改动之后知道自己有没有把系统打坏
3. 可以做更系统的实验，而不是靠直觉调参数

### 问题 30：为什么 RAG 项目一定要做 ablation？

因为 RAG 是多环节系统。

query rewrite、chunking、融合、rerank、prompt、回答模式切换，这些模块都可能让系统变复杂，但不一定真的带来收益。没有 ablation，很难判断你加的复杂度是不是值得。

所以这类项目的一个成熟信号，就是它不只会做功能，也会问：

“这个功能值不值得存在？”

## 六、面试官如果连续深挖，你可以怎么接

### 连续追问 1：为什么不是纯向量检索？

第一层回答：

因为企业文档里很多问题对结构和术语极度敏感，纯向量容易“意思像，但不够准”。

第二层追问：那为什么不只做全文检索？

可以答：

因为很多真实问题会换表达方式，全文检索会漏掉语义相近但措辞不同的内容，所以 dense 语义召回仍然必要。

第三层追问：那你们到底在平衡什么？

最终可以收口到：

本质是在平衡“词项精确命中”和“语义泛化能力”，所以才会做 dense+sparse 加上结构信号，而不是押注单一路径。

### 连续追问 2：chunk 为什么不用 token-based？

第一层回答：

当前实现优先考虑的是稳定性和统一处理多源文档，所以先用了字符滑窗。

第二层追问：字符和 token 明显不一致，那是不是不合理？

可以答：

它不是最理想，但在工程上足够可控。尤其是在文档来源复杂、格式不一的阶段，先用透明的切法把主链做稳，是合理取舍。

第三层追问：那你会怎么升级？

可以答：

先统一当前两套切片逻辑，再引入 token-aware 控制，最后再考虑 semantic chunking。因为不先统一策略，后面的优化很容易被系统内不一致吞掉。

### 连续追问 3：你们怎么降低 hallucination？

第一层回答：

不是单靠 prompt，而是检索质量、证据分级、风险控制和引用约束四层一起工作。

第二层追问：那如果检索结果本身错了呢？

可以答：

这套系统不能彻底防住错误证据，但它至少做了两件事：尽量让更可靠的证据排前面，以及不在弱证据场景下强行 grounded 回答。

第三层追问：所以你们已经解决幻觉了吗？

最稳的回答是：

没有，我不会这么讲。我们做的是显著降低风险、提高可追溯性，但还没到 verifier 驱动的最终答案校验。

### 连续追问 4：为什么要做 common knowledge 路径？

第一层回答：

因为不是所有问题都值得直接拒答，有些问题在证据不足时仍然可以用低风险的常识方式回答。

第二层追问：那这不就是绕过 RAG 了吗？

可以答：

不是绕过，而是显式降级。关键是要把边界说清楚，不要把这类回答伪装成“来自知识库证据”。

第三层追问：那怎么避免用户误解？

可以收口到：

这就是为什么 answer mode、引用和检索信息很重要，它们让系统能够区分“有证据的回答”和“无证据的降级回答”。

## 七、项目外也很容易被问到的通用题

### 问题 31：如果不看这个项目，你怎么选 embedding 模型？

可以这样答：

我会从四个维度看：

1. 语种和领域适配  
   中文、英文、代码、法律、客服，这些场景适合的 embedding 不一样。

2. 质量和成本  
   更大的模型通常效果更好，但索引成本、查询成本和部署成本都更高。

3. 延迟和吞吐  
   如果系统要频繁重建索引，embedding 吞吐就很关键。

4. 和检索链的配合方式  
   如果我本来就打算做 dense+sparse hybrid，那 dense 模型不一定要追求最大，而要追求和 sparse 信号组合后的整体收益。

然后再补一句和本项目呼应的话：

在 `rag-qa` 里我就不是把模型单独看，而是放在“Qdrant hybrid 检索”这个整体里看。

### 问题 32：什么情况下你会说“不适合用 RAG”？

这个题很常见，也很容易答概念。

更好的回答是：

如果问题的答案并不主要来自外部知识，而是来自复杂推理、交易执行、实时状态或者严格规则系统，那 RAG 可能不是主解法，只能是辅助。比如：

1. 强事务型操作
2. 高实时状态判断
3. 复杂多步决策执行
4. 需要强确定性、不能接受模糊生成的场景

RAG 最适合的是“答案主要来源于外部知识，但用户问法和知识表达不完全一致”的场景。

### 问题 33：你怎么看 prompt injection？

一个成熟回答应该包含三点：

1. 这不是 prompt 本身的问题，而是“把不可信输入和可信规则混在一起”的系统问题。
2. 不能只靠一句 system prompt“请忽略恶意指令”来解决。
3. 更合理的做法是多层防御，包括输入检查、上下文治理、工具权限边界和输出约束。

然后可以再落回项目：

在这个项目里，已经做的部分主要是风险识别和回答模式控制，这是一层实用防线，但不是完整安全体系。

### 问题 34：你怎么看 offline eval 和 online eval 的关系？

可以这样答：

offline eval 的价值是让你能稳定比较版本，防止改动把系统明显打坏；online eval 的价值是让你知道用户真实分布下系统到底表现如何。

两者不能互相替代：

1. 只有 offline，没有真实反馈，可能会高估效果
2. 只有 online，没有回归门槛，迭代会非常危险

这个问题答得好的关键，是体现你把 eval 当成工程体系的一部分，而不是论文实验。

### 问题 35：如果让我从零设计一个 RAG 系统，你会先做哪三件事？

这是很典型的项目外问题，但非常适合借当前项目回答。

我会说：

1. 先把知识边界和数据来源梳理清楚  
   不先搞清楚知识从哪里来、怎么更新、怎么删除，后面的系统很容易变脏。

2. 先做一个稳定、可解释的基础检索链  
   不急着堆复杂算法，先让切片、索引、检索、引用这些基本环节跑稳。

3. 从第一天就做评测和回归  
   因为 RAG 是多环节系统，没有评测，你根本不知道自己是在优化还是在制造复杂度。

## 八、终极版面试收口话术

如果面试官最后问你：“你觉得这个项目最能体现你能力的是什么？”

你可以这样答：

我觉得最能体现我能力的，不是我接了某个模型或者用了某个向量库，而是我理解这类 AI 应用真正难的地方不在单点能力，而在整条链路。比如知识怎么统一接入、为什么要 worker 化、为什么不能只做向量检索、为什么回答前要先判断证据边界、为什么要做评测和回归。这些点在 `rag-qa` 里不是孤立功能，而是被接成了一套完整系统。它当然还有提升空间，比如 chunking、缓存和最终答案验证都可以更强，但我认为它最有价值的地方，是已经具备了真实 AI 应用该有的工程骨架。
