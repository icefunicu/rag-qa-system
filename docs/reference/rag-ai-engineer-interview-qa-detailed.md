# RAG-QA AI 应用工程师面试深度问答版

> 说明
>
> - 这份材料不是项目总结，也不是代码导读，而是“面试时怎么回答”的详细版。
> - 目标是让一个原本不了解项目的技术人员，看完以后能真正理解这个系统的主链路、关键参数、做法、权衡和边界。
> - 所有内容都只基于已经确认过的实现。已经做过的会直接说，没明确做过的会明确标成“下一步优化方向”，不会混写。

## 一、先把项目讲明白

### 问题 1：这个项目到底是什么？一句话怎么讲？

最短版本可以这样回答：

这是一个工程化的企业知识问答系统，前面有完整的知识入库链，后面有完整的在线问答链，不是“上传 PDF 然后做一次向量检索”的 demo。

如果面试官愿意听 30 秒到 1 分钟，我会这么展开：

这个项目分成两个服务。一个是知识库服务，负责把不同来源的数据统一接进来，再做解析、切片、OCR、索引和知识库内检索；另一个是 API 网关，负责处理用户聊天、跨知识库聚合检索、答案生成、会话持久化和预算控制。真正有技术含量的地方，不在于“能不能调模型”，而在于它把数据接入、检索、证据约束、回答生成、稳定性和评测都串成了一条完整链路。

如果我想再说得更像工程项目，而不是产品介绍，我会补一句：

它的重点不是把模型变得多聪明，而是让回答更有证据、更可控、链路更稳、上线后更容易验证和回归。

### 问题 2：这个项目的主链路是什么？

这个问题最好分成两段讲，一段讲知识入库，一段讲在线问答。

#### 2.1 知识入库流

知识入库大概是这样走的：

1. 用户或者系统把文档送进来。来源不只有文件上传，还包括本地目录、Notion 页面、网页抓取结果和 SQL 查询结果。
2. 系统先做接入侧治理，比如判断是不是新文档、是不是更新版本、旧版本要不要软删除、要不要创建新的入库任务。
3. 真正的重活不是在接口里同步做，而是交给异步 worker。这样大文档不会卡住 API，请求也更容易恢复。
4. worker 拿到任务后会先下载原始内容，再根据文件类型决定如何解析。文本、PDF、DOCX、图片都支持，但路径不完全一样。
5. 解析后的内容会被拆成 section 和 chunk，然后连同标题、页码、字符范围、来源类型这些元数据一起写入结构化存储。
6. 如果文档里有图片、扫描页或者图文混排内容，还会补一轮 OCR，把视觉内容也变成可检索的文本单元。
7. 最后再把 section 和 chunk 写进向量索引，供后面的检索链使用。

这个流程的核心不是“把文档存进去”，而是“把原始知识转换成后面可检索、可引用、可追踪的证据单元”。

#### 2.2 在线问答流

在线问答这条链也不要讲成“用户提问 -> 模型回答”这么粗。

更真实的讲法是：

1. 用户提问先进入网关。
2. 网关会做会话处理，比如读取最近历史、校验请求幂等、检查预算、确定这次问题的知识范围。
3. 检索不是直接扔给一个向量库，而是先由网关把问题 fanout 到一个或多个知识库。
4. 每个知识库内部会做一套比较完整的检索编排：先改写问题，再做结构检索、全文检索和向量检索，之后做融合和重排。
5. 网关把各个知识库返回的证据再统一裁剪，控制每个库、每份文档和整次回答的证据上限。
6. 拿到证据之后，系统不会默认模型一定可以回答，而是先判断证据够不够、风险高不高，再决定走“基于证据的回答”“常识降级回答”还是直接拒答。
7. 回答生成完成后，不只是返回一段文本，还会把引用、检索信息、调用痕迹和使用量一起保存下来。

最值得强调的一点是：

这个系统的问答主链不是“检索一下然后拼到 prompt 里”，而是“先找到候选证据，再做证据选择，再决定回答边界”。

### 问题 3：为什么说它不是普通 demo？

因为普通 demo 往往只有三步：

1. 上传一个文件
2. 做一次向量检索
3. 把结果丢给模型

而这个项目至少多了下面几层：

1. 多源知识接入，不是只有本地 PDF。
2. 异步 worker 入库，不是请求里同步跑全链。
3. 混合检索，不是单一路径向量搜索。
4. 有证据强弱判断，不是检索完就一定答。
5. 有 prompt 安全和回答边界控制。
6. 有会话预算、幂等、限流、重试和降级。
7. 有评测、消融和回归门禁，不是“感觉有效”。

所以更准确的说法是：

它已经不是“能跑起来的 RAG”，而是“能治理、能验证、能上线”的 RAG 雏形。

### 问题 4：这个项目最有技术含量的部分是什么？

如果只能挑一个点，我会选混合检索链。

原因很简单：

它不是一个库的简单调用，而是一整套检索思路。系统会先对 query 做处理，再让结构命中、全文命中和语义召回一起工作，之后再用融合和重排把证据筛到比较适合放进上下文的程度。这里面既有搜索思路，也有参数控制，还有和生成阶段的联动。

如果能讲两个点，我会再加上“回答前的证据约束”。

因为很多人讲 RAG 都停在“怎么召回”，但真实系统里更难的一件事其实是：什么时候该答，什么时候只能保守回答，什么时候应该拒答。这个项目至少把这件事显式做了，而不是默认交给模型自由发挥。

### 问题 5：这个项目目前偏基础的地方是什么？

这个问题必须敢讲，不然很容易被问穿。

我会直接说三个点：

1. 文本切片还是比较务实的字符滑窗，不是 token-aware，也不是 semantic chunking。
2. 缓存和 embedding 优化做得不深，更多是控制链路边界，而不是有一套复杂缓存体系。
3. 回答可靠性虽然做了证据分级和 prompt 安全，但还不是 verifier 级的最终答案校验。

这样讲的好处是：

1. 真实，不会吹过头。
2. 你反而能显得更懂工程边界。
3. 后面如果被追问“那你会怎么优化”，你就有很自然的展开空间。

## 二、知识入库链怎么做的

### 问题 6：知识是怎么进系统的？

这个问题不要回答成“支持上传和同步”，那样太浅。

更完整的回答应该是：

系统支持多种知识源，包括普通文件上传、本地目录、Notion 页面、网页内容和 SQL 查询结果。它们进入系统以后，并不是各走一套完全不同的后处理，而是会尽量统一成“待入库文档”的形式，后面再交给同一条 ingest 链处理。

为什么这点重要？

因为企业知识天然分散。如果每种来源都做一套独立问答逻辑，后面会出现两个问题：

1. 检索层没法统一。
2. 文档版本、删除、增量更新和引用都很难管。

这个项目的做法更像“先把来源统一，再统一治理，再统一检索”。

### 问题 7：为什么入库要异步做，而不是同步接口完成？

这个问题很适合体现系统观。

可以这样答：

因为真实的知识入库并不是简单保存文件，它后面至少还有几件事：下载原始内容、解析正文、切片、写结构化存储、做 OCR、写向量索引、更新状态。如果把这些都放在同步接口里，接口会变得很慢，而且任何一步失败都会让整个请求体验很差。

所以这个项目用的是异步 worker。它的价值主要有四个：

1. 上传接口快，不需要等完整入库结束。
2. 长任务更容易做重试和恢复。
3. 可以把文档处理拆成多个阶段，而不是非得一次做完。
4. 文档可以先“部分可用”，不用等所有增强任务完成。

这个“部分可用”是一个很值得讲的点。

因为系统不是等 OCR、视觉增强、所有索引都结束了才让文档能查，而是优先让主文本链先可用。它背后的工程判断是：很多时候“先能查，再增强”比“强一致等全部完成”更合理。

### 问题 8：文档解析到底做了什么？

这里要把“解析”和“切片”分开讲。

解析阶段做的事情，本质上是把原始文件变成后面能理解的文本结构。这个项目明确支持文本、PDF、DOCX 和图片等格式。不同格式的路径不一样，但目标一致：抽出可读内容，尽量识别章节或标题结构，形成 section 级内容。

这里有一个非常关键的工程点：

系统不是只保留正文字符串，而是会连带保留结构信息。也就是说，一段内容来自哪个文档、哪个章节、哪个页码、在原文里的什么范围，这些信息后面都能继续传下去。

为什么这个点重要？

因为后面的检索、引用和回答都依赖这些结构信息。如果一开始只保留纯文本，后面就很难做“这是哪一页、哪一段、哪一章”的追踪。

### 问题 9：文本切片是怎么做的？

这是最容易被面试官追问的部分，必须讲细。

当前系统采用的是比较务实的字符滑窗切片，不是 token 切片，也不是语义切片。核心参数是：

1. 主窗口大约是 `1000` 个字符。
2. overlap 在 `100` 到 `120` 个字符之间。
3. 先按 section 结构切，再在每个 section 内做滑窗。

这个设计怎么理解？

可以把它看成一个典型的工程折中。

如果 chunk 太小，比如几百字符，确实会更精细，但很容易把一个完整语义切碎，结果就是召回到了很多“局部相关但解释不完整”的片段。  
如果 chunk 太大，比如两三千字符，虽然上下文更完整，但噪声会迅速增大，后面的 embedding、检索、rerank 和 prompt 都会更贵。

所以当前这套参数明显是在取一个中间值：

1. 让 chunk 不至于太碎。
2. 让每个 chunk 又不会大到把很多弱相关信息一起塞进去。

那 overlap 为什么要有？

因为 chunk 边界最容易切断完整语义。尤其是长句子、实体关系、定义和解释分布在边界附近的时候，如果完全不重叠，下一步召回很可能只拿到“半句正确”。适当 overlap 的作用，是让边界附近的信息不会一下子被截断。

但我会明确承认两个局限：

1. 这不是按 token 控制的，所以和模型实际上下文开销不是完全一致。
2. 这是规则式切法，不理解真正的语义边界。

如果面试官问“为什么不是 semantic chunking”，更稳的回答是：

因为当前项目优先追求的是统一多源文档、稳定入库和整体链路可控。语义切片值得做，但在现阶段，低复杂度的字符滑窗更容易落地，也更容易调试。

### 问题 10：切片之后保留了哪些信息？为什么要保留？

这个问题经常被忽略，但其实很关键。

当前这套系统在 chunk 和 section 层面保留的信息比较全，至少包括：

1. 所属文档
2. 文档标题
3. section 标题
4. 字符范围
5. 页码
6. 来源类型，比如正文还是 OCR
7. 视觉资源相关标识，比如 asset 或缩略图关联

为什么要保留这些元数据？

因为后面至少有四处会用到：

1. 检索时做过滤
2. 融合和重排时区分来源类型
3. prompt 里组织证据时给模型更多上下文
4. 最终回答里返回引用和来源

所以可以这么总结：

这个项目不是把 chunk 当成一堆匿名文本，而是把 chunk 当成“带出处的证据单元”。

### 问题 11：OCR 和视觉增强是怎么接进来的？

这个点不要讲成“我们做了多模态”，容易过头。

更准确的说法是：

系统支持从 PDF、DOCX 和图片里提取视觉资产，再做 OCR，把识别出的文字和视觉区域变成新的 section / chunk，然后和正文一起进入检索链。

这意味着什么？

意味着系统不是只依赖纯文本正文。对于扫描件、图文混排内容或者原本正文抽取不完整的页面，至少还能通过 OCR 把信息补进来。

这个能力的价值，是“扩大可覆盖知识范围”，不是“做成多模态大模型系统”。

你最好主动补一句 trade-off：

OCR 一定会带来噪声，所以这一步的收益是把原本会丢的信息尽量保住，但它不可能替代更强的版面理解和表格结构恢复。

### 问题 12：向量索引是怎么做的？

这里不要直接说“用了 Qdrant”，太平。

更好的说法是：

系统把切好的 section 和 chunk 都写进 Qdrant，走的是 dense+sparse 混合索引。dense 负责语义相似，sparse 更偏关键词和词项匹配。这样做的好处是，不用在“只做关键词”和“只做语义”之间二选一，而是让两种表达形式一起参与检索。

当前明确能说的参数包括：

1. 向量集合名是 `kb-evidence`
2. dense embedding 用的是 `BAAI/bge-small-zh-v1.5`
3. sparse 模型用的是 `Qdrant/bm25`

为什么这样设计？

因为这个项目面对的是中文企业知识场景。这里更像是一个偏稳妥的工程选型：

1. 选一个比较轻量、可部署的中文 embedding 模型
2. 再用 sparse 信号补词项命中
3. 最终在同一套检索链路里做 hybrid

这个设计最大的好处不是“模型最先进”，而是“索引和检索表达力更完整”。

## 三、在线问答链怎么做的

### 问题 13：用户提问之后，系统第一步做什么？

第一步不是去调模型，也不是直接去查向量库。

系统先做的是“把这次问题放回到一个可控的会话和知识范围里”。具体包括：

1. 识别这次请求属于哪个会话
2. 读取最近几轮历史
3. 检查幂等，避免同一个请求被重复执行
4. 检查并发限制，避免某个用户或整个系统被打爆
5. 检查本轮会话预算，避免多轮聊天成本失控
6. 解析这次问题到底应该查哪些知识库、哪些文档范围

这段虽然不直接体现“AI 算法”，但特别体现 AI 应用工程师的价值。

因为真实的 AI 系统不是单次调用，而是会话、权限、范围、预算和质量一起要控。

### 问题 14：检索到底是怎么跑的？

这个问题必须讲得细，因为它是整套系统的核心。

建议按三层讲。

#### 第一层：网关先做跨知识库聚合

网关不会自己做向量检索，它更像一个调度层。它先根据当前问题的范围，把请求 fanout 到一个或多个知识库。这里的意义是：系统支持跨知识域查询，而不是把所有内容粗暴塞到一个大索引里。

这样设计有两个明显好处：

1. 知识边界更清楚，不同 corpus 可以独立治理。
2. 检索结果可以按库级别做聚合和裁剪。

当前这层有几个明确参数：

1. fanout 并发上限默认是 `4`
2. 每个知识库请求下游时，先拿 `8` 条候选
3. 聚合后再裁剪成：每个 corpus 最多 `3` 条、每个文档最多 `2` 条、总数最多 `8` 条

这几个数字其实透露了一个很重要的设计思路：

系统并不追求“召回越多越好”，而是从一开始就在控制最终可进入 prompt 的证据规模。

#### 第二层：单个知识库内部做混合检索

单个知识库内部会先做 query rewrite，再跑三类检索：

1. 结构检索
2. 全文检索
3. 向量检索

结构检索适合章节、标题、目录感很强的问题。  
全文检索适合术语、参数、短语精确命中。  
向量检索适合语义表达不同但意思相近的问题。

也就是说，它不是在赌某一种检索器永远最好，而是让不同信号互相补位。

#### 第三层：融合和重排

三路召回以后，并不是简单拼起来，而是先融合，再重排。

当前明确的融合权重是：

1. structure：`1.3`
2. fts：`1.0`
3. vector：`0.9`

这个权重组合其实能看出设计偏好：

系统更信任结构性命中，其次是全文，向量排在第三。这个不是说向量不重要，而是说明在当前知识形态下，标题、章节和术语仍然是强信号。

向量候选先取 `40` 条，不是最终都给模型，而是先给后面的融合和重排留候选池。最后 rerank 候选池大小大约是 `max(limit*3, 12)`，也就是先给一个稍大的池子，再压成最终可用结果。

### 问题 15：为什么还要重排？融合还不够吗？

可以这样讲：

融合解决的是“不同检索器的结果怎么合成一份排序”，但它本质还是信号层的合并。重排解决的是另一个问题：对于已经进到候选池里的内容，query 和 chunk 之间谁更贴近，谁只是看起来相关但其实没那么适合回答。

所以这两者不是替代关系，而是前后两层：

1. 融合负责把不同来源的候选拉到一个空间里
2. 重排负责把真正适合回答的证据往前推

当前系统的重排默认更偏 heuristic，也支持更强的外部服务，但更稳的讲法一定是：

系统真实接了 rerank 能力，但更强的 cross-encoder 是否是默认线上路径，要看具体部署配置，不能一概而论。

### 问题 16：query rewrite 在这里的价值是什么？

这个问题不要回答成“为了提升召回率”，太空。

更具体的说法是：

query rewrite 的价值是把用户问题从“自然提问方式”变成“更适合检索的表达”。比如：

1. 去掉一些疑问词和口语成分
2. 提取问题的 focus
3. 做词项扩展
4. 让结构检索和全文检索更容易命中

它的本质不是改写得更漂亮，而是改写得更利于检索器工作。

如果面试官问“为什么不用原问题直接检索”，你可以说：

原问题更适合给模型理解，不一定适合给搜索系统检索。这两个目标并不完全一致。

### 问题 17：为什么还要在网关侧再做一轮裁剪？

因为单个知识库内最相关，不等于跨知识库全局最相关。

而且最终进入模型上下文的内容一定是有限的，所以需要在更高一层再做一次预算控制。当前这层控制很明确：

1. 每个知识库最多保留 3 条
2. 每个文档最多保留 2 条
3. 总证据数最多 8 条

这种做法的优点是简单、稳定、好解释。缺点是规则味比较重，还不是智能上下文管理。

所以更高级但仍然真实的讲法是：

当前实现已经意识到了“多召回不等于好回答”，但上下文管理仍然偏规则式，不是 query-aware 的动态裁剪。

### 问题 18：prompt 是怎么设计的？

不要讲成“我写了一个很强的 prompt”。

更成熟的讲法是：

这个系统把 prompt 管理做成了独立能力，而不是散落在路由代码里。它至少区分了两类回答模式：

1. 基于证据的 grounded prompt
2. 常识类的 common knowledge prompt

这个区分很重要，因为两类回答的边界不一样。grounded 回答必须能和证据对应起来，而 common knowledge 本质上是一种显式降级，不应该假装自己来自知识库。

另外，prompt 里塞进去的也不是一堆原始 chunk 文本，而是会把证据整理成结构化信息，包括：

1. 来自哪个知识库
2. 来自哪个文档
3. 来自哪个 section
4. 页码和字符范围
5. 当前证据分数
6. 关键引用片段
7. 必要时带上压缩后的原文

也就是说，系统不是简单做“文本拼接”，而是在给模型喂“带来源信息的证据块”。

### 问题 19：系统怎么决定“能答、降级答、还是拒答”？

这正是这套系统比较有工程感的地方。

它不是默认所有问题都强行回答，而是会先判断证据状态。你可以把它理解成三种路径：

1. 证据足够，走 grounded answer
2. 证据不够，但问题风险低，走 common knowledge 降级回答
3. 风险高或者证据太弱，直接 refusal

这背后的工程思路很值得讲：

真实系统里最危险的不是“答不出来”，而是“看起来很自信地答错”。所以这个项目明显更偏向在边界不清楚的时候保守一点。

这里还有一层 prompt safety，用来检查提示注入、越权引导、强行要求去掉引用之类的问题。它的价值不是让系统更聪明，而是让边界更稳。

### 问题 20：回答生成阶段的关键参数是什么？怎么理解这些参数？

这个问题讲参数时不要铺天盖地，只讲最关键的几个。

当前比较重要的生成参数有：

1. grounded 回答温度大约是 `0.2`
2. grounded 回答最大输出大约限制在 `1200` tokens 以内
3. common knowledge 回答温度大约是 `0.4`
4. common knowledge 使用的历史也更严格，只保留较少的历史条数和字符数

这些参数的含义其实不复杂：

1. grounded answer 的目标是稳，不是发散，所以温度偏低
2. 证据型回答不希望输出特别长，否则容易稀释引用关系
3. common knowledge 本来就不是强证据回答，所以会稍微放一点发散性

如果面试官问“为什么不是更高温度”，你可以很自然地说：

因为这个系统的目标不是创作，而是尽量在证据约束下稳定回答，低温度更符合这个目标。

### 问题 21：引用是怎么做的？为什么这个点重要？

这里一定不要夸成“我们做了完整的事实核验体系”。

更稳的讲法是：

系统会把证据来源一路带到回答侧，并要求最终回答里带引用标记。也就是说，回答不是一段脱离来源的自然语言，而是尽量能指回对应的证据单元。

这个点的重要性在于两件事：

1. 对用户来说，回答更可追溯
2. 对系统自己来说，后续更容易做评测、调试和问题定位

但这里也要诚实说边界：

当前更像“引用约束和引用标记”，不是严格的最终答案逐句对齐校验。它能提升可追溯性，但不能等价成“已经彻底解决事实一致性问题”。

## 四、性能、成本、稳定性和工程边界

### 问题 22：这个项目怎么做性能优化？

这个问题不要回答“做了很多优化”，一定要具体。

当前系统比较实在的优化，不在于某个炫技型算法，而在于把几个关键边界先锁住：

1. 入库侧用异步 worker，避免长任务堵住接口
2. section 和 chunk 写入时做 batch，降低数据库开销
3. 检索侧限制候选规模，比如向量候选先取 40，而不是无限放大
4. 网关侧限制跨库 fanout 并发，默认大约是 4
5. 聚合后再次限制每个知识库、每个文档和整体证据数量
6. 模型侧限制最大输出长度和会话预算

所以这套系统的优化思路更像：

先把最容易失控的地方掐住，而不是先上一个复杂缓存体系。

这是一种很典型的工程取舍。它的优点是可控、容易定位问题，缺点是上限不会特别高。

### 问题 23：这个项目有缓存吗？

这个问题要非常小心。

更真实的回答是：

有基础设施层面的复用，比如模型实例、向量库客户端、向量存储对象会做复用；但如果你问的是“有没有成熟的 query cache 或 retrieval cache”，那从当前主链路看并不算强。

所以我更建议这样说：

这个项目当前的性能策略主要不是靠缓存，而是靠控制请求规模、控制 fanout、控制证据数量和控制会话预算。缓存不是完全没有，但也不是这套系统最强的地方。

### 问题 24：这个项目怎么控制成本？

可以从三个维度讲。

第一，检索侧不做无边界召回。  
第二，生成侧控制输出长度和历史长度。  
第三，会话层面有预算概念，不让单个会话无限消耗。

这个回答的重点不是“我们算得很精确”，而是“我们意识到成本是一个系统级问题，不只是模型单价问题”。

为什么这么说？

因为在 RAG 里，真正烧钱的不只是模型调用本身，还包括：

1. 召回太多导致上下文变大
2. 多轮历史越来越长
3. 跨知识库 fanout 越来越多

所以如果你只盯着模型单次价格，很多时候是控制不住总成本的。

### 问题 25：这个项目怎么做稳定性保障？

这部分非常适合拉开你和“只会调 SDK”的人的差距。

可以拆成四层讲：

#### 第一层：请求幂等

避免同一次问题因为重试、网络抖动或前端重复提交而被执行多次。

#### 第二层：并发限制

系统对全局和单用户并发都有控制，不让某个用户或者某一波流量把整个问答链路拖垮。

#### 第三层：长任务重试和恢复

入库 worker 有明确的 lease、重试退避和死信处理。失败不是直接丢掉，而是按类别决定重试还是终止。

当前明确的重试退避大概是：

`5 -> 15 -> 45 -> 135 -> 300` 秒

这说明它不是简单的“失败就重来”，而是有节奏的退避。

#### 第四层：下游失败降级

无论是跨知识库 fanout，还是模型调用路由，都不是“一处失败全盘崩”。检索层接受部分知识库失败但整体继续，模型层也支持在特定错误场景下切换到备用 route。

如果你想让这段更有力量，可以补一句：

这套系统没有追求“绝不失败”，而是在做“失败时怎么别把整条链一起带崩”。

### 问题 26：模型路由和重试怎么理解？

这里也要讲真话。

当前系统不是那种“SDK 自动重试很多次”的风格，而是更偏显式路由和显式 fallback。也就是说，它更愿意把切换模型这件事当成明确策略，而不是藏在 SDK 内部的黑盒重试里。

这样做的好处是：

1. 行为更可控
2. 更容易知道到底是哪个路由失败了
3. 不容易因为过度自动重试把问题掩盖掉

代价是：

如果网络临时抖动，而 fallback 条件又比较保守，你可能会更依赖外围稳定性设计。

### 问题 27：这个项目最真实的 trade-off 是什么？

我会回答三个最核心的 trade-off。

#### 第一，召回率和上下文噪声之间的 trade-off

多信号检索能提升召回，但也更容易把噪声带进 prompt。所以系统必须在融合、重排和裁剪之间做平衡。

#### 第二，可用性和强一致之间的 trade-off

入库链路选择让文档先部分可用，再逐步增强，而不是所有流程都做完才允许查询。

#### 第三，工程复杂度和算法深度之间的 trade-off

当前系统先把多源接入、worker、混合检索、回答边界和评测做实了，但在 semantic chunking、缓存体系、最终答案验证这些点上还没有走到特别深。

这一段是非常适合体现成熟度的，因为它说明你理解：系统设计从来不是“越复杂越好”，而是“复杂度花在哪儿最值”。

## 五、如果面试官继续深挖，你可以这样答

### 问题 28：如果让你重做一版，你最先优化哪三件事？

这个问题不能回答得太飘，要基于当前代码现实。

我会说三件事，而且顺序很明确。

#### 第一，统一 chunking 策略

因为当前切片虽然可用，但仍然是字符滑窗，而且有两处参数实现并不完全一样。这个问题会影响 embedding、检索、上下文和成本，是投入产出比最高的优化点。

#### 第二，升级重排策略

现在已经有 rerank 能力，但默认更偏 heuristic。下一步我会让更强的语义重排变得更稳定，同时保留轻量级方案作为降级路径。

#### 第三，做更智能的上下文预算控制

现在的证据裁剪是有效的，但偏规则。下一步更值得做的是 query-aware 的上下文管理，比如根据问题类型动态决定召回规模、保留哪些证据、历史保留多少。

### 问题 29：如果面试官问“你们为什么没做更高级的方案”，怎么答才不虚？

你可以这样说：

我不回避这个问题。当前实现确实没有把 semantic chunking、verifier、完整缓存体系这些都做进去。但从工程角度看，我更认可先把主链打实：多源接入、异步入库、混合检索、证据约束、会话治理和评测回归，这些是让系统“真的能跑、能控、能迭代”的基础。更高级的方案一定值得做，但我不认为在主链还不稳的时候先堆最复杂算法是好决策。

这段回答为什么强？

因为它不是为没做找借口，而是在讲“为什么当前阶段优先级这么排”。

### 问题 30：怎么把这个项目讲得高级，但又不夸大？

最稳的模板是这样的：

1. 先承认它不是算法创新型 RAG，而是工程化 RAG。
2. 再强调它真正做实的部分：多源入库、混合检索、证据约束、worker、评测回归。
3. 最后明确边界：chunking、缓存、verifier 还可以继续加强。

你会发现，这种讲法反而更像高级工程师。

因为高级不是“把所有东西都说成自己做了”，而是“知道哪些地方已经足够硬，哪些地方还在演进”。

## 六、最后给一段“看完就能讲”的长回答

如果面试官说“你完整讲讲这个项目”，可以直接用下面这段：

这个 `rag-qa` 项目我会把它理解成一个工程化的企业知识问答系统，而不是普通 RAG demo。它前面有一条完整的知识入库链，支持文件、本地目录、Notion、网页和 SQL 等多种知识源。数据进来以后不会在接口里同步跑完，而是先做统一治理，再交给异步 worker 处理。worker 负责解析正文、做字符滑窗切片、保留文档标题、章节、页码和字符范围这些元数据；如果文档里有图片或扫描页，还会补一轮 OCR，把视觉内容也并进索引。最后再把 section 和 chunk 写进向量库，形成后面可检索的证据单元。

问答这边，用户先进入网关，网关会处理会话、幂等、预算和知识范围，再把问题 fanout 到对应的知识库。单个知识库内部不是只做向量检索，而是先改写问题，再同时跑结构检索、全文检索和语义检索，之后再做融合和重排。这个地方我认为是整套系统最有技术含量的部分，因为它体现的是检索工程，而不是单一库调用。拿到候选证据之后，系统不会默认模型一定回答，而是先判断证据强弱和 prompt 风险，再决定是按证据回答、按常识降级回答，还是直接拒答。最后输出里还会保留引用和调用痕迹，方便追溯和评测。

如果评价这套系统，我会说它做得比较实的是主链工程化：多源入库、异步 worker、混合检索、证据边界控制、稳定性治理和评测回归。这些都是普通 demo 不太会认真处理的部分。它的不足也很明确，比如 chunking 还是字符滑窗，缓存体系不算强，回答最终校验也没有做到 verifier 级别。所以我会把它讲成一个已经有生产思维的 RAG 系统，而不是一个追求炫技算法的项目。
