# RAG-QA AI 应用工程师面试口语版材料

> 说明
>
> - 这是一份“口语版”面试材料，重点是讲得顺、讲得稳、讲得像真人，不是代码导读。
> - 内容仍然严格基于已经确认过的实现，只是尽量少堆代码路径。
> - 凡是项目里没明确做过的，我会单独放到“别讲过头”的部分。

## 1. 先用一句话认识这个项目

如果要先用一句话介绍这个项目，我会说：

这是一个偏工程化的企业知识问答系统，不是单纯的“上传 PDF 然后向量检索”。它把知识入库、检索编排、回答边界控制、会话治理和评测回归都接成了一条比较完整的链路。

再展开一点，可以这样说：

这个项目分成两块。前面一块是知识入库，负责把文件、网页、Notion、SQL 结果这些不同来源的数据统一转成可检索内容；后面一块是在线问答，负责把用户的问题送进检索链，再把证据组织好交给模型生成回答。真正有价值的地方，不是“能不能答”，而是“答的时候有没有证据、边界清不清楚、系统稳不稳、出了问题能不能追得回来”。

## 2. 面试开场怎么讲

### 2.1 30 秒版本

我做的这个 `rag-qa` 项目，不是单一脚本或者本地 demo，而是拆成了知识库服务和 API 网关。知识库侧负责多源文档接入、解析、切片、OCR 和向量索引，网关侧负责跨知识库检索聚合、基于证据的回答生成和会话治理。它最核心的地方不是只做向量检索，而是把混合检索、证据约束、安全控制和评测回归做进了主链路，所以更像一个可落地的 AI 应用系统。

### 2.2 1 分钟版本

这个项目的主链路其实很清楚，可以分成入库和问答两段。入库这边支持多种数据源，不只是上传文件，还包括本地目录、Notion、网页和 SQL。数据进来之后，不会同步卡在接口里，而是进入异步 worker，做解析、切片、OCR 增强和向量索引。问答这边，用户的问题先进入网关，网关会处理会话、范围和历史，再去调知识库做检索。检索不是单一路径，而是结构检索、全文检索、向量检索一起跑，再做融合和重排。回答阶段也不是让模型自由发挥，而是先看证据够不够、风险高不高，再决定是走基于证据的回答、常识回答，还是直接拒答。我的理解是，这个项目的强项在于工程化比较完整，弱项是切片和缓存这类算法细节还偏务实，没有做到特别花哨。

### 2.3 3 分钟版本

如果让我完整讲这个项目，我会从“它解决的不是一个单点问题，而是一条完整链路”开始。

第一段是知识入库。这个系统接的不是单一文件源，而是多种异构数据，包括上传文件、本地目录、Notion 页面、网页内容和 SQL 查询结果。它们进来之后会做统一治理，而不是各走各的逻辑。然后任务会被交给异步 worker 处理，worker 负责把原始内容转成 section 和 chunk，再做 OCR 和视觉增强，最后把文本和元数据一起写进索引。这一段最像真实工程，而不是 demo，因为它考虑了长任务、失败重试、状态推进和部分可用。

第二段是在线问答。用户进来以后，系统不会直接把问题丢给模型，而是先做范围解析、历史处理和检索。检索层最值得讲，因为它不是只做向量搜索，而是把结构命中、全文命中和语义召回一起用，再做融合和重排。这样做的原因很简单，企业文档里有很多标题、章节、术语和参数，只靠语义相似很容易“像，但不准”。

第三段是回答生成。这个项目有一个我比较认可的点，就是它在生成前会先判断证据是否足够，再决定是否允许模型按知识库证据来回答。如果证据弱或者有 prompt 风险，它不会硬答，而是走降级路径或者拒答。也就是说，它在努力把“回答能力”和“回答边界”同时做出来。

如果要评价这套系统，我会说它最像生产系统的地方，是多源入库、worker 化处理、混合检索、证据约束和评测回归；最偏基础的地方，是 chunking 还属于字符滑窗，缓存和最终答案验证还不够深。

## 3. 最值得讲的 6 个点

### 3.1 混合检索，而不是纯向量检索

这个点是我最建议拿来做主亮点的。

最自然的说法是：

我们没有把检索做成单一路径，而是把结构检索、全文检索和向量检索放在一起用。原因是不同问题需要的信号不一样。有些问题带明显章节词、制度名、字段名，这种时候结构和关键词命中很重要；有些问题换了表达方式，这时候向量检索更有价值。最后再把不同信号融合，并加一层重排，把真正适合进 prompt 的证据往前放。

为什么这个点值得讲：

因为它一下子就能把你和“只会用向量库”的候选人区分开。你不是在讲一个库的 API，而是在讲“我怎么设计一个更稳的召回链”。

面试官最喜欢追问的地方通常有三个：

1. 为什么不只做向量检索？
2. 为什么还需要重排？
3. 这样会不会太慢？

这三个问题的回答可以统一成一个思路：

纯向量检索在企业文档里不够，因为很多信息本来就强依赖标题、章节和关键词；重排的价值是把融合后的候选再压一轮噪声；性能上则靠候选数量限制和分层处理来控，不是无限召回。

你可以直接背的一段话是：

我在这个项目里最认可的一点，是检索不是单一路径。我们把结构、全文和向量三种信号一起用，再做融合和重排。这样做不是为了“更复杂”，而是因为真实知识库里同时存在章节结构、术语精确命中和语义相似三类需求。它的代价是链路更长、参数更多，但好处是比纯向量方案更稳。

### 3.2 不是“检索完就回答”，而是先判断能不能答

这个点特别适合回答“怎么降低幻觉”。

更口语的讲法是：

很多 RAG 系统的问题不在于检索不到，而在于检索到一点内容之后模型就开始补全。我们这套系统在生成前会先看证据够不够，再决定是按证据回答、按常识降级回答，还是直接拒答。这个思路本质上是在给模型划边界，而不是单纯让模型更聪明。

它为什么重要：

因为面试官不太相信“我 prompt 写得很好”这种答案，但会认可“我对回答边界做了显式控制”。这更像工程能力。

如果被追问“这样会不会太保守”，可以这么答：

会，这就是 trade-off。保守会损失一点覆盖率，但在企业问答里，答错往往比答不出更危险，所以这个系统明显是偏可靠性优先的设计。

可以直接背的一段话是：

我们不是检索完就让模型自由发挥，而是先对证据强度做分级，再结合 prompt 风险判断，决定这次回答到底应该走哪种模式。这样做的好处是，系统不是把所有问题都包装成“我有把握回答”，而是把可信边界说清楚。

### 3.3 入库做成了异步 worker，而不是同步接口

这个点很有工程味。

最自然的讲法是：

入库这件事在真实系统里不可能只是“收个文件然后立刻完成”。它后面还有解析、切片、OCR、向量化、索引更新这些步骤，所以我们把它放进异步 worker，而不是卡在上传接口里。这样用户不会因为一份大文档等很久，系统也更容易做失败重试和状态恢复。

这个点为什么强：

因为它说明你在思考系统生命周期，而不是只关心 happy path。尤其是“长任务怎么恢复”“失败之后怎么处理”“什么时候能先查到部分结果”，这些都很像真实场景。

如果被追问“为什么要分阶段可用”，你可以说：

因为文本主链通常比视觉增强快，先让文档尽快可查，比强行等所有增强任务做完更合理。这本质上是在做“渐进式可用”，而不是强一致。

### 3.4 多源接入不是堆功能，而是统一知识入口

这一点不要讲成“我们接了很多源，好厉害”，要讲成“为什么这对 AI 应用有意义”。

更好的说法是：

这个系统支持多种知识源，本质上不是为了功能列表好看，而是因为企业知识本来就分散在不同地方。真正重要的是，它们不是分别做一套逻辑，而是最终都被统一成后面的入库链路，统一进文档治理、切片和索引。这就意味着，后面的检索和回答层不需要针对每种来源重新发明一套协议。

如果被追问“多源接入最大的难点是什么”，你可以答：

不是接进来本身，而是接进来之后怎么统一元数据、版本、删除和增量更新。否则知识会越来越脏。

### 3.5 OCR 和视觉增强，让系统不只会读纯文本

这个点很适合讲出差异化，但不要吹成“多模态大模型系统”。

更稳的说法是：

很多企业文档并不是标准文本 PDF，而是扫描件、截图或者图文混排。如果只做正文抽取，很多信息在入库时就丢了。这个项目做了一步比较实用的增强，就是把视觉内容先通过 OCR 转成文本，再并入现有检索链路。它的价值不是把系统变成炫技型多模态应用，而是让原本会掉的信息尽量还能进入证据链。

如果被追问“这样会不会引入噪声”，你可以直接承认：

会，所以这一步的价值是扩覆盖，不是保证绝对准确。视觉内容进来之后仍然要靠后面的检索排序和回答约束去兜底。

### 3.6 评测和回归门禁，让优化不是拍脑袋

这是一个非常适合面试加分的点。

最好这样讲：

很多 RAG 项目做到最后，大家都觉得自己在优化，但其实没人能证明改动有没有带来真实收益。这个项目比较好的地方是，它不是只做功能，还做了评测、消融对比和回归门禁。也就是说，像 query rewrite、融合、rerank 这种东西，不是“感觉有用”就一直留着，而是有机会被对比和被回归约束。

这个点的高级表达方式是：

我不会说“我们指标提升了多少”，因为我不想虚构结果。但我会说，这个项目已经把“能不能证明它更好”这件事写进了工程流程里。这一点本身就很重要。

## 4. 高频追问怎么回答

### 4.1 为什么不用纯向量检索？

可以直接回答：

因为企业文档里很多问题对关键词和结构很敏感。比如制度名称、表字段、章节标题、参数项，这些场景只靠语义相似不稳。纯向量检索的优点是泛化好，但缺点是容易“像，但不准”。所以我们更偏向把向量能力当成一类信号，而不是全部。

### 4.2 chunk 为什么这么切？

可以这样说：

当前实现是比较务实的字符滑窗，核心思路是在上下文完整性和 token 成本之间取平衡。切得太小，信息碎；切得太大，噪声和成本上升。它的优点是稳定、可控、容易统一多种文档源，缺点是语义边界感知不强。所以我会把它讲成“当前阶段的工程折中”，不会包装成高级切片算法。

### 4.3 overlap 为什么需要？

你可以说：

overlap 的目的不是多塞点内容，而是降低 chunk 边界把完整语义切断的风险。尤其是长句子、实体关系或者跨段说明，完全不重叠时更容易出问题。当然 overlap 太大也会带来冗余，所以本质还是平衡。

### 4.4 为什么选这个 embedding 方案？

更稳的说法是：

当前选择明显是偏“中文可用、成本可控、方便做 hybrid 检索”的工程路线，不是追求理论最强模型。对一个要长期跑的系统来说，这种选择很常见。我的重点不会放在“这个模型一定最好”，而是放在“它怎么和后面的 sparse 检索、向量库和总体成本结构配合”。

### 4.5 为什么选 Qdrant？

你可以这么讲：

因为这套系统不只是要做相似度搜索，还要做知识域隔离、文档级过滤和 hybrid 检索。Qdrant 在这方面比较直接，尤其适合把 dense 和 sparse 放在一套链路里做。这里我会强调“这是当前主链的实现”，而不是泛泛谈论各种向量库优劣。

### 4.6 你们怎么控制 token 成本？

可以答：

这套系统没有靠特别复杂的缓存来省钱，而是优先控制边界。比如限制召回候选数量、限制跨库聚合后的证据总量、限制会话预算、限制生成长度。也就是说，它的成本控制更像“别让链路无限膨胀”，而不是“已经做了一个很强的成本优化平台”。

### 4.7 你们怎么降低 hallucination？

最稳的说法是：

我不会把它说成“彻底防住”，而是说它做了几层控制。第一层是检索质量，尽量把相关证据找准；第二层是证据分级，不让弱证据问题默认走强回答；第三层是 prompt 风险控制；第四层是引用标记和可追溯输出。也就是说，它不是只在 prompt 里说一句“不要编”，而是从检索到生成一起做约束。

### 4.8 这个项目最大的不足是什么？

可以答得诚实一点：

我觉得最大的不足有三个。第一，chunking 还是比较基础，是字符滑窗，不是更高级的语义切片。第二，缓存和 embedding 优化不算深，更多是边界控制。第三，回答准确性控制更多靠规则和 prompt 约束，还没有到 verifier 级别。这些都不影响项目真实价值，但如果面试里不主动承认，反而更容易被问穿。

## 5. 如果要讲“我的工程判断”，怎么讲得高级

这一部分很适合面试最后收口。

你可以这样说：

我对这个项目最认可的地方，不是它用了哪些热门名词，而是它做了几个比较克制但正确的工程判断。比如它没有一开始就追求最复杂的 chunking，而是先用稳定的滑窗把多源入库、混合检索和回答链路打通；它没有把所有问题都丢给模型，而是先做证据分级和风险控制；它也没有只停在功能层，而是补了评测和回归。换句话说，这个项目的价值不在于某一个点特别炫，而在于整条链是可运行、可治理、可验证的。

这段话的好处是：

1. 它不会吹得太过。
2. 它能体现你的系统观。
3. 它能把“偏基础的实现”解释成“阶段性的合理取舍”，而不是能力不够。

## 6. 哪些地方不要讲过头

这一段很重要，面试里很多人不是答错，而是答太满。

### 6.1 不要讲成高级切片系统

更稳的说法：

当前是比较务实的字符滑窗，优点是稳定和易控，缺点是语义感知有限。

不要说：

我们做了 semantic chunking、parent-child retrieval、token-aware 优化。

因为这些从当前代码里并不能强支撑。

### 6.2 不要把缓存讲得太重

更稳的说法：

我们做了基础复用和边界控制，但没有看到特别完整的 query cache 体系。

不要说：

我们做了很成熟的缓存架构。

### 6.3 不要把 OCR 讲成完整多模态

更稳的说法：

我们把视觉内容先转成文本，再并进原有 RAG 链路，这是一种实用增强。

不要说：

我们已经做了端到端多模态问答。

### 6.4 不要把外部 rerank 讲成默认主路径

更稳的说法：

系统支持更强的重排能力，但当前默认路径更偏 heuristic，外部能力是否启用取决于环境。

### 6.5 不要把回答可靠性讲成“已经彻底解决”

更稳的说法：

我们做了证据约束、风险控制和引用约束，明显比普通 demo 更稳，但还不是最终验证器体系。

## 7. 最后给你的“可直接背诵版”

如果你只想背一段最稳的版本，可以用下面这段：

我做的这个 `rag-qa` 项目，本质上是一个工程化的企业知识问答系统，而不是简单的向量检索 demo。它前面有完整的知识入库链，支持多源数据接入、异步处理、OCR 增强和索引更新；后面有完整的问答链，会先做混合检索和重排，再根据证据强弱和风险决定怎么回答。这个项目最有技术含量的地方，是它没有把 RAG 理解成“检索一下然后喂给模型”，而是把检索质量、回答边界、系统稳定性和评测回归一起考虑了。它的弱点我也会明确说，就是切片和缓存还偏基础，但这反而让我更容易把它讲得真实，因为我知道哪些是已经做实的，哪些是下一步优化方向。
