# RAG-QA 项目面试深度材料

## 第 1 部分：项目核心架构总览

这个 `rag-qa` 项目是一个完整的企业级 RAG 问答系统，核心架构分为两条主链路：

1. **知识入库流**：从文件上传开始，经过文档解析（支持 TXT、PDF、DOCX）、基于滑动窗口的文本切片（window=1000, overlap=100）、向量生成（支持本地和外部 embedding provider），最终存储到 Qdrant 向量库，同时建立结构化索引和全文索引。
2. **问答调用流**：用户提问经过查询重写，生成查询向量，然后通过多信号检索（结构检索、全文检索、向量检索）获取候选结果，使用加权 RRF 算法融合，再经过重排序，最后结合上下文和历史生成 prompt，调用 LLM 生成带引用的回答。

项目最有技术含量的部分是**多信号检索融合机制**，通过结构、全文和向量的三重检索，并使用加权 RRF 算法融合结果，显著提高了检索准确性。

目前实现偏基础的部分包括**评估机制**（只有脚本，未集成到系统）和**可观测性**（仅基础日志，缺乏完整监控）。

这份材料如果用于讲当前最新版后端，建议把“LangGraph 编排升级”放到最前面讲。因为它不只是换了一个框架，而是把系统从“命令式串流程”升级成了“显式状态机 + 可恢复运行时”。

## 第 2 部分：项目亮点

### 新增亮点：从手写 LangChain 编排升级为 LangGraph Agentic RAG 运行时

1. **解决的问题**：
   - 旧的问答链路虽然已经能完成检索、生成、重试和审计，但主流程更多依赖 service 层手写推进。
   - 这种实现对于简单 happy path 没问题，但一旦加入人工澄清、可恢复执行、节点级追踪、运行中断、幂等 resume，状态会迅速分散。
   - 最终会出现一个典型工程问题：系统能跑，但很难清楚说明“这次 run 现在在哪个节点、为什么停在这里、恢复后会从哪继续”。
2. **这次到底“清掉”了什么**：
   - 不是把 LangChain 生态整体删掉。
   - 保留了底层仍然有价值的能力，比如 `Document`、retriever、模型调用、共享工具函数。
   - 真正被清掉的是“由业务函数自己顺手维护状态机”的编排方式。
   - 现在查询面的主编排改由 LangGraph 负责，节点、边、checkpoint、interrupt 都变成显式对象。
3. **Gateway 图怎么拆节点**：
   - `prepare_turn`：负责读取会话、规范化请求、解析 scope、拉历史消息、执行检索准备，并判断当前是否要进入人工澄清。
   - `human_review_turn`：负责调用 `interrupt(...)` 暂停运行，把结构化问题交给外部；恢复后把人工提交的问题改写或权限选择写回 payload。
   - `generate_answer`：负责真正的 grounded answer 生成，并补上 verification 信息，例如回答里有没有正确带引用标记。
   - `persist_turn`：负责把稳定结果写入聊天消息存储，避免中途状态还没确定就提前落消息。
4. **为什么 Gateway 要这样拆**：
   - `prepare_turn` 是分支决策点，所有“要不要继续答”的判断都应该在这里收口。
   - `human_review_turn` 把暂停和恢复做成统一机制，而不是业务侧额外维护一套挂起状态。
   - `generate_answer` 单独拆开后，可以明确区分“准备阶段失败”还是“生成阶段失败”。
   - `persist_turn` 放在最后，能保证中断 run 不会写出半成品消息。
5. **KB 检索图怎么拆节点**：
   - `prepare_request`：把自然语言问题改写成真正的检索请求，同时解析实际允许参与检索的文档范围。
   - `run_signal_retrievers`：统一执行结构检索、全文检索、向量检索，并记录 degraded signals 和 warnings。
   - `fuse_and_rerank`：把三路候选统一成证据块，做加权 RRF 融合，再执行重排，最后输出统一的 `RetrievalResult`。
6. **为什么 KB 也要 graph 化**：
   - 如果只有 Gateway 图，KB 对上层仍然是黑盒，上层只能知道“我发了一次 retrieve”，却不知道 retrieval 内部经历了哪些阶段。
   - 现在 KB 也有了明确的阶段边界，后续无论要扩更多检索信号、并行召回，还是做失败降级，都有稳定骨架可以挂载。
7. **状态与恢复怎么做**：
   - LangGraph checkpoint 是主状态源，真正决定 run 停在哪、恢复后从哪继续。
   - `chat_workflow_runs` 不再承担主恢复逻辑，而是退化成投影和审计层，供列表、筛选、运营分析使用。
   - `chat_graph_interrupts` 专门记录人工介入请求与回复，解决的是“给用户看什么”和“用户提交了什么”，不是替代 checkpoint。
8. **这件事的工程价值**：
   - 让运行态显式化。当前问题、证据、答案、人工介入、当前节点都能直接读到。
   - 让恢复语义稳定化。恢复不再是“从大概这个阶段重试一下”，而是从 graph checkpoint 精确继续。
   - 让审计可读化。一个 run 是在 `prepare_turn` 被挂起，还是在 `generate_answer` 失败，能直接回答。
   - 让测试边界清晰化。现在可以单独测 interrupt/resume、graph projection、retrieval graph，而不必每次都用整链黑盒测试。
9. **关键文件/函数**：
   - `apps/services/api-gateway/src/app/gateway_graph.py`
   - `apps/services/api-gateway/src/app/gateway_chat_graph_routes.py`
   - `apps/services/api-gateway/src/app/gateway_workflows.py`
   - `apps/services/knowledge-base/src/app/retrieve.py`
   - `apps/services/knowledge-base/src/app/kb_query_routes.py`
10. **为什么值得讲**：
    - 很多 RAG 项目只会讲“用了向量检索和大模型”，但讲不清运行时怎么组织、失败后怎么恢复、人工介入怎么接进来。
    - 这个亮点的价值在于，它体现的是后端工程能力，而不只是算法拼装能力。
    - 对 AI 工程岗、Agent 平台岗、RAG 后端岗来说，这比单纯讲“用了哪个 embedding 模型”更能体现架构水平。
11. **面试表述**：
    - “我这次做的不是简单把 LangChain 链写长一点，而是把查询面的编排层从手写 service 逻辑升级成 LangGraph 运行时。Gateway 侧把一次问答拆成 prepare、human review、generate、persist 四个稳定节点，KB 侧把 retrieval 拆成 query rewrite、三路召回、融合重排三个阶段。这样之后，中断、恢复、审计、节点级测试都有统一语义，系统从能跑提升到了能解释、能恢复、能演进。” 
12. **可能的追问**：
    - 追问 1：“为什么不继续用 LangChain Runnable 或者手写 if/else，而要换 LangGraph？”
    - 追问 2：“为什么 Gateway 和 KB 都要 graph 化？”
    - 追问 3：“你们的主状态源到底是什么？workflow_run 还是 checkpoint？”
    - 追问 4：“节点为什么只拆这么几个，不拆得更细？”
    - 追问 5：“这次升级最直接的业务收益是什么？”
13. **追问回答**：
    - 追问 1：“因为问题已经不是‘能不能把流程串起来’，而是‘能不能稳定地暂停、恢复、追踪和审计’。Runnable 很适合把几个步骤接起来，但对 interrupt、checkpoint 和显式状态机支持没有 LangGraph 这么直接。继续手写 if/else 当然也能做，但状态和恢复语义会越来越散。” 
    - 追问 2：“因为如果只有 Gateway graph 化，KB 仍然是黑盒，上层只能看到一次 RPC，看不到 retrieval 内部经历了 query rewrite、三路召回、融合重排这些阶段。要做真正可解释的 Agentic RAG，上下游都要有清楚的阶段边界。” 
    - 追问 3：“现在主状态源是 LangGraph checkpoint。`workflow_run` 保留，但主要承担投影和审计职责。也就是说，恢复依赖 checkpoint，列表展示和运营查询依赖投影表。” 
    - 追问 4：“节点不是越多越好。我的原则是一个节点只做一类稳定职责。比如 Gateway 的 `prepare_turn` 是天然决策点，`generate_answer` 是天然执行点，`persist_turn` 是天然落库点。再往下拆会让图过碎，反而不利于理解和维护。” 
    - 追问 5：“最直接的收益是中断恢复和人工澄清终于有了统一机制。之前这种能力要靠业务层硬拼，现在它是运行时原生能力。同时 run 的状态更透明，排障和测试成本也明显下降。” 

### 亮点 1：多信号检索融合机制

1. **解决的问题**：单一检索方式（如仅向量检索）存在局限性，无法兼顾结构信息、关键词匹配和语义相关性
2. **具体实现**：
   - 同时使用三种检索信号：结构检索（匹配标题和章节）、全文检索（基于 PostgreSQL FTS）、向量检索（基于 Qdrant 混合检索）
   - 通过加权 RRF（Reciprocal Rank Fusion）算法融合三种信号的结果
3. **关键文件/函数**：
   - `retrieve.py` 中的 `_run_signal_retrievers` 函数负责并行执行三种检索
   - `_fuse_and_rerank` 函数实现 RRF 融合和重排序
4. **关键参数**：
   - 融合权重：`FUSION_WEIGHTS = {"structure": 1.3, "fts": 1.0, "vector": 0.9}`
   - 最终返回结果数：`limit=8`
5. **为什么值得讲**：实现了工业级 RAG 系统常用的多信号检索策略，能有效提高检索准确性
6. **trade-off**：
   - 优点：结合了多种检索方式的优势，提高了召回率和准确性
   - 缺点：增加了系统复杂度和检索延迟
7. **当前局限**：权重固定，未实现动态调整；重排序算法相对简单
8. **优化方向**：
   - 基于用户反馈动态调整信号权重
   - 引入更复杂的重排序模型（如 BERT-based reranker）
   - 实现自适应的检索信号选择
9. **面试表述**："我们的系统实现了多信号检索融合机制，同时使用结构检索、全文检索和向量检索，通过加权 RRF 算法融合结果。这种设计能兼顾不同检索方式的优势，比如结构检索能快速定位相关章节，全文检索对关键词匹配敏感，向量检索能捕捉语义相关性，最终显著提高了检索准确性。"
10. **可能的追问**：
    - 追问 1："为什么选择这三种检索信号？"
    - 追问 2："权重是怎么确定的？"
    - 追问 3："RRF 算法的原理是什么？"
    - 追问 4："如何并行执行这三种检索？"
    - 追问 5："检索延迟如何？"
11. **追问回答**：
    - 追问 1："我们选择这三种信号是因为它们覆盖了不同的检索维度：结构信号能利用文档的自然层级关系，全文信号对关键词匹配敏感，向量信号能捕捉语义相关性。这三种信号互补，结合起来能提高检索的全面性和准确性。"
    - 追问 2："权重是基于经验和初步测试确定的，结构信号权重最高（1.3）是因为标题和章节信息通常最能反映内容主题，全文信号次之（1.0），向量信号最低（0.9）。后续可以基于用户反馈和 A/B 测试动态调整。"
    - 追问 3："RRF 算法的核心思想是将每个文档在不同检索结果中的排名转换为分数，公式为 1/(rank + k)，其中 k 是超参数（我们用的是 60），然后按信号权重加权求和。这种算法能有效融合不同来源的排序结果，对召回率和准确率都有提升。"
    - 追问 4："三种检索是串行执行的，但每个检索内部都有优化，比如向量检索使用了 Qdrant 的混合检索，全文检索使用了 PostgreSQL 的 FTS 索引。后续可以考虑将它们并行化，进一步降低延迟。"
    - 追问 5："目前端到端检索延迟在 500ms 左右，主要受向量检索和重排序影响。我们通过限制每个信号的候选结果数（如向量检索返回 40 条）来控制延迟。"

### 亮点 2：基于滑动窗口的文本切片策略

1. **解决的问题**：长文档需要合理切片才能有效存储和检索，切片质量直接影响检索和生成效果
2. **具体实现**：
   - 使用固定滑动窗口切片，窗口大小 1000 字符，重叠 100 字符
   - 先按文档结构（标题）分割章节，再对每个章节进行切片
3. **关键文件/函数**：
   - `parsing.py` 中的 `_chunk_sections` 函数
4. **关键参数**：
   - window=1000
   - overlap=100
5. **为什么值得讲**：文本切片是 RAG 系统的基础，直接影响后续所有环节的效果
6. **trade-off**：
   - 优点：实现简单，可控性强，能保证上下文连续性
   - 缺点：未考虑语义完整性，可能将完整语义分割到不同 chunk
7. **当前局限**：仅基于字符数，未考虑 token 数或语义边界
8. **优化方向**：
   - 实现基于语义的切片（如使用 sentence transformer 或 LLM 分割）
   - 支持动态 chunk size 调整
   - 考虑 token 数而非仅字符数
9. **面试表述**："我们实现了基于滑动窗口的文本切片策略，窗口大小 1000 字符，重叠 100 字符。首先按文档结构分割章节，再对每个章节进行切片，这样能保证 chunk 内部的上下文连续性，同时通过重叠部分避免完整语义被分割。"
10. **可能的追问**：
    - 追问 1："为什么选择 1000 字符的窗口大小？"
    - 追问 2："overlap 为什么设为 100？"
    - 追问 3："为什么不使用基于语义的切片？"
    - 追问 4："切片后如何处理 metadata？"
    - 追问 5："对检索和生成效果有什么影响？"
11. **追问回答**：
    - 追问 1："1000 字符是综合考虑了上下文完整性和 token 成本的结果。太小的 chunk 会丢失上下文，太大的 chunk 会增加 token 消耗，1000 字符大概对应 250-300 个英文 token，适合大多数场景。"
    - 追问 2："100 字符的 overlap 是为了避免将完整的语义单元分割到不同 chunk，比如一个长句子或一个概念。overlap 过大会增加冗余，过小则可能丢失上下文联系。"
    - 追问 3："目前使用固定窗口切片主要是因为实现简单、性能高，适合初期版本。基于语义的切片虽然效果更好，但计算成本高，后续可以考虑集成。"
    - 追问 4："每个 chunk 都会保留完整的 metadata，包括文档 ID、章节标题、页码、字符范围等，这些 metadata 在检索时用于过滤和排序，在生成回答时用于添加引用。"
    - 追问 5："合理的切片策略能提高检索召回率，因为每个 chunk 主题更集中，同时能减少生成时的 token 消耗，提高回答质量。我们通过测试发现，1000 字符窗口在召回率和成本之间取得了较好的平衡。"

### 亮点 3：基于 Qdrant 的混合检索实现

1. **解决的问题**：单一向量检索在处理关键词匹配和低频词时效果不佳
2. **具体实现**：
   - 使用 Qdrant 的混合检索模式，同时支持稠密向量和稀疏向量
   - 稠密向量使用 BAAI/bge-small-zh-v1.5 模型生成
   - 稀疏向量使用 Qdrant/bm25 模型生成
3. **关键文件/函数**：
   - `qdrant_store.py` 中的 `_get_vector_store` 函数配置混合检索
4. **关键参数**：
   - 向量维度：512（稠密向量）
   - 距离 metric：余弦相似度
   - 检索模式：RetrievalMode.HYBRID
5. **为什么值得讲**：混合检索是当前 RAG 系统的主流趋势，能兼顾语义相关性和关键词匹配
6. **trade-off**：
   - 优点：结合了稠密向量的语义理解和稀疏向量的关键词匹配优势
   - 缺点：增加了索引大小和计算复杂度
7. **当前局限**：稀疏向量模型固定，未支持自定义或动态调整
8. **优化方向**：
   - 支持稀疏向量模型的动态切换
   - 实现稠密向量和稀疏向量的权重调整
   - 考虑引入更多类型的向量（如多模态向量）
9. **面试表述**："我们使用 Qdrant 实现了混合检索，同时利用稠密向量和稀疏向量。稠密向量捕捉语义相关性，稀疏向量处理关键词匹配，两者结合能显著提高检索准确性，尤其是在处理专业术语和低频词时效果更好。"
10. **可能的追问**：
    - 追问 1："为什么选择 Qdrant 作为向量库？"
    - 追问 2："混合检索的权重如何设置？"
    - 追问 3："稠密向量和稀疏向量分别是怎么生成的？"
    - 追问 4："混合检索的性能如何？"
    - 追问 5："如何处理向量库的索引更新？"
11. **追问回答**：
    - 追问 1："选择 Qdrant 主要是因为它原生支持混合检索，性能优秀，部署简单，同时提供了丰富的 API 和 SDK。相比其他向量库，Qdrant 在混合检索和稀疏向量支持方面表现突出。"
    - 追问 2："目前使用 Qdrant 默认的混合检索权重，后续可以通过 API 调整稠密向量和稀疏向量的权重比例，根据业务场景优化。"
    - 追问 3："稠密向量使用 BAAI/bge-small-zh-v1.5 模型生成，这是一个轻量级但效果优秀的中文 embedding 模型；稀疏向量使用 Qdrant 内置的 bm25 模型生成，基于词频和逆文档频率计算。"
    - 追问 4："混合检索的性能表现良好，单查询延迟在 100ms 左右，我们通过限制向量库的索引大小和使用缓存机制进一步优化了性能。"
    - 追问 5："向量库索引更新采用批量方式，当文档更新时，会删除旧的向量索引，重新生成并插入新的向量索引，确保索引与文档内容保持一致。"

### 亮点 4：LLM 调用的模型路由和降级机制

1. **解决的问题**：单一 LLM 可能存在稳定性问题，不同模型在不同场景下表现各异
2. **具体实现**：
   - 基于配置的模型路由，支持根据 prompt 类型选择不同模型
   - 实现了模型调用失败后的自动降级机制

### 亮点 5：带引用的回答生成机制

1. **解决的问题**：纯生成式回答缺乏可信度，用户无法验证信息来源
2. **具体实现**：
   - 检索结果作为证据添加到 prompt 中
   - LLM 生成回答时自动引用相关证据
   - 后处理阶段确保引用标记正确
3. **关键文件/函数**：
   - `grounded_answering.py` 中的 `ensure_citation_markers` 函数
   - `gateway_answering.py` 中的 `generate_grounded_answer` 函数
4. **关键参数**：
   - 证据数量：最多 8 条
   - 引用格式：[1], [2] 等数字标记
5. **为什么值得讲**：带引用的回答能显著提高用户信任度，是企业级 RAG 系统的重要特性
6. **trade-off**：
   - 优点：提高了回答的可信度和可验证性
   - 缺点：增加了 prompt 长度和 token 消耗
7. **当前局限**：引用标记依赖 LLM 生成，偶尔会出现错误或遗漏
8. **优化方向**：
   - 实现更严格的引用验证机制
   - 支持多种引用格式
   - 实现引用与原文的精确对应
9. **面试表述**："我们实现了带引用的回答生成机制，检索结果作为证据添加到 prompt 中，LLM 生成回答时会自动引用相关证据，最后通过后处理确保引用标记正确。这种方式能显著提高回答的可信度，方便用户验证信息来源。"

    - 追问 1："引用是如何生成的？"

    - 追问 3："如果 LLM 生成的引用有误怎么办？"

## 第 3 部分：项目技术难点/技术挑战

### 难点 1：检索结果的噪声过滤

1. **本质难在哪里**：检索结果中不可避免地会包含与查询无关或相关性较低的文档，这些噪声会影响后续的回答生成质量
2. **为什么是真实工程问题**：在大规模知识库中，检索噪声是普遍存在的，直接影响最终回答的准确性和可信度
3. **项目中目前的处理方式**：
   - 通过多信号检索提高初始检索准确性
   - 使用加权 RRF 融合减少噪声
   - 对融合结果进行重排序
4. **代码依据**：
   - `retrieve.py` 中的 `_fuse_and_rerank` 函数
5. **现有方案的不足**：
   - 重排序算法相对简单
   - 未考虑文档间的相关性
   - 缺乏基于用户反馈的动态优化
6. **优化方向**：
   - 引入更复杂的重排序模型（如 BERT-based reranker）
   - 实现基于上下文的相关性过滤
   - 支持基于用户反馈的检索优化
7. **可能的追问**：
   - 追问 1："检索噪声主要来自哪些方面？"
   - 追问 2："如何量化检索噪声？"
   - 追问 3："重排序算法的效果如何评估？"
   - 追问 4："有没有考虑使用检索增强的生成方式？"
8. **追问回答**：
   - 追问 1："检索噪声主要来自三个方面：一是向量检索的语义偏差，二是全文检索的关键词匹配歧义，三是结构检索的标题匹配不准确。"
   - 追问 2："目前通过人工评估检索结果的相关性来量化噪声，后续计划引入自动评估指标（如 NDCG、MAP 等）。"
   - 追问 3："重排序效果通过对比重排序前后的检索准确性来评估，初步测试显示能将相关文档的平均排名提升 20% 左右。"
   - 追问 4："目前使用的是先检索后生成的方式，后续可以考虑引入检索增强的生成方式，如 Retrieval-Augmented Generation (RAG) 的进阶版本。"
9. **诚实表达**："目前我们的噪声过滤机制还比较基础，主要依赖多信号融合和简单的重排序。虽然能解决大部分问题，但在处理复杂查询或大规模知识库时，噪声问题仍然存在。我们正在考虑引入更先进的重排序模型和基于用户反馈的优化机制。"

### 难点 2：token 成本控制

1. **本质难在哪里**：RAG 系统的 token 成本主要来自两部分：embedding 生成和 LLM 调用。随着知识库规模和查询量的增加，成本会迅速上升
2. **为什么是真实工程问题**：在生产环境中，token 成本是 RAG 系统的主要运营成本之一，必须进行有效控制
3. **项目中目前的处理方式**：
   - 对查询向量进行缓存（LRU 缓存，最大 512 个）
   - 限制 prompt 中的证据数量（最多 8 条）
   - 限制 LLM 生成的最大 token 数（1200）
4. **代码依据**：
   - `embeddings.py` 中的 `_embed_query_text_cached` 函数实现查询向量缓存
   - `gateway_answering.py` 中的 `generate_grounded_answer` 函数限制证据数量和最大 token 数
5. **现有方案的不足**：
   - 缓存机制简单，未考虑缓存失效策略
   - 证据数量固定，未根据查询复杂度动态调整
   - 缺乏成本监控和预警机制
6. **优化方向**：
   - 实现更智能的缓存机制，支持基于使用频率的缓存失效
   - 动态调整证据数量和 LLM 生成的最大 token 数
   - 引入成本监控和预警系统
   - 考虑使用更高效的模型或量化技术
7. **可能的追问**：
   - 追问 1："token 成本主要来自哪些环节？"
   - 追问 2："缓存命中率如何？"
   - 追问 3："如何平衡成本和回答质量？"
   - 追问 4："有没有考虑使用更经济的模型？"
8. **追问回答**：
   - 追问 1："token 成本主要来自两个环节：一是 embedding 生成（尤其是外部 API），二是 LLM 调用（包括 prompt 和生成的回答）。其中 LLM 调用通常占总成本的 80% 以上。"
   - 追问 2："目前查询向量缓存的命中率在 30% 左右，主要受查询多样性影响。后续可以通过优化缓存策略（如增加缓存大小、改进缓存键设计）来提高命中率。"
   - 追问 3："我们通过以下方式平衡成本和质量：限制证据数量但保证覆盖主要信息，调整温度参数控制回答的创造性和准确性，根据不同场景选择合适的模型。"
   - 追问 4："是的，我们考虑过使用更经济的模型，比如针对特定任务微调的小型模型，或者开源模型部署在本地。但目前为了保证回答质量，主要使用商业模型，后续会根据成本和质量的平衡进行调整。"
9. **诚实表达**："目前我们的 token 成本控制还比较基础，主要通过简单的缓存、固定的证据数量和最大 token 数限制来控制。虽然能满足当前需求，但在大规模部署时可能会面临成本压力。我们正在考虑引入更智能的成本控制机制，包括动态调整参数和使用更高效的模型。"

### 难点 3：降低幻觉风险

1. **本质难在哪里**：LLM 生成回答时可能会出现幻觉（hallucination），即生成的内容与事实不符或没有证据支持
2. **为什么是真实工程问题**：幻觉会严重影响 RAG 系统的可信度，尤其是在企业级应用中，错误的信息可能导致严重后果
3. **项目中目前的处理方式**：
   - 严格限制 LLM 只能基于提供的证据生成回答
   - 在 prompt 中明确要求 LLM 不要编造信息
   - 实现带引用的回答，方便验证
4. **代码依据**：
   - `grounded_answering.py` 中的 `chat_prompt_messages` 函数构建严格的 prompt
5. **现有方案的不足**：
   - 依赖 LLM 的自我约束，效果有限
   - 缺乏独立的事实核查机制
   - 未实现幻觉检测和修正
6. **优化方向**：
   - 引入独立的事实核查模块
   - 实现幻觉检测算法
   - 对可疑回答进行降级处理
7. **可能的追问**：
   - 追问 1："如何定义和检测幻觉？"
   - 追问 2："prompt 中具体是怎么限制 LLM 的？"
   - 追问 3："有没有统计过幻觉率？"
   - 追问 4："除了 prompt 设计，还有哪些降低幻觉的方法？"
8. **追问回答**：
   - 追问 1："幻觉定义为生成的内容与提供的证据不符或没有证据支持。目前主要通过人工评估检测，后续计划引入自动检测算法，如对比生成内容与证据的语义相似度。"
   - 追问 2："在 prompt 中我们明确要求：'请严格基于提供的证据回答问题，不要添加任何额外信息，不要编造事实。如果无法从证据中找到答案，请明确说明。' 同时在证据部分清晰标记每个证据的编号和内容。"
   - 追问 3："目前还没有系统的幻觉率统计，主要通过随机抽样人工评估，初步估计幻觉率在 5% 左右。后续计划建立完善的评估体系。"
   - 追问 4："除了 prompt 设计，还可以通过以下方式降低幻觉：使用更精确的模型、减少 prompt 长度、增加证据的相关性、实现事实核查机制等。"
9. **诚实表达**："目前我们主要通过 prompt 设计来降低幻觉风险，要求 LLM 严格基于提供的证据生成回答，并实现了带引用的回答机制。虽然能有效降低幻觉率，但仍然依赖 LLM 的自我约束。我们正在考虑引入更主动的幻觉检测和事实核查机制，进一步提高回答的准确性。"

## 第 4 部分：重点深挖专题

### 专题 1：文本切片策略

1. **实现位置**：`parsing.py`
2. **入口与调用链**：
   - `parse_document` → `_parse_text_sections`/`_parse_docx` → `_chunk_sections`
3. **完整流程**：
   - 读取文件内容
   - 按文档结构（标题）分割章节
   - 对每个章节使用滑动窗口切片
   - 生成 KBChunk 对象列表
4. **关键参数**：
   - window=1000（字符）
   - overlap=100（字符）
   - 标题匹配正则：`TXT_HEADING_RE` 和 `DOCX_HEADING_RE`
5. **为什么这么设计**：
   - 基于字符数的固定窗口切片实现简单，性能高
   - overlap 确保完整语义不被分割
   - 先按结构分割再切片能保留文档的层级关系
6. **替代方案**：
   - 基于 token 的切片：更精确，但计算成本高
   - 基于语义的切片：效果更好，但实现复杂
   - 基于段落的切片：更自然，但可能导致 chunk 大小不均匀
7. **trade-off**：
   - 简单性 vs 语义完整性
   - 性能 vs 切片质量
   - 固定大小 vs 动态调整
8. **面试官可能怎么追问**：
   - "chunk size 为什么选 1000 字符？"
   - "overlap 为什么设为 100？"
   - "为什么不使用基于语义的切片？"
   - "切片后如何处理 metadata？"
9. **标准回答**：
   - "chunk size 选 1000 字符是综合考虑了上下文完整性和 token 成本。1000 字符大概对应 250-300 个英文 token，能容纳完整的语义单元，同时不会导致 prompt 过长。"
   - "overlap 设为 100 字符是为了避免将完整的语义单元分割到不同 chunk，比如一个长句子或一个概念。overlap 过大会增加冗余，过小则可能丢失上下文联系。"
   - "目前使用固定窗口切片主要是因为实现简单、性能高，适合初期版本。基于语义的切片虽然效果更好，但计算成本高，后续可以考虑集成。"
   - "每个 chunk 都会保留完整的 metadata，包括文档 ID、章节标题、页码、字符范围等，这些 metadata 在检索时用于过滤和排序，在生成回答时用于添加引用。"
10. **当前成熟度评价**：基础

9. **标准回答**：

   - "embedding 维度 512 是综合考虑了效果和性能的结果。更高的维度可能提供更好的语义表示，但会增加存储和计算成本。512 维度在大多数场景下能取得较好的平衡。"
   - "查询向量缓存使用 Python 的 lru_cache 实现，最大缓存 512 个查询。缓存键包含模型配置和查询文本，确保不同配置下的查询不会冲突。"
   - "对于长文本，我们先进行切片，然后对每个切片生成 embedding。在检索时，会考虑所有相关切片的向量。"
10. **当前成熟度评价**：中等

4. **关键参数**：

### 专题 4：检索召回策略

1. **实现位置**：`retrieve.py`
2. **入口与调用链**：
   - `retrieve_kb_result` → `_prepare_request` → `_run_signal_retrievers` → `_fuse_and_rerank`
3. **完整流程**：
   - 处理查询，进行查询重写
   - 并行执行三种检索：结构检索、全文检索、向量检索
   - 使用加权 RRF 融合三种检索结果
   - 对融合结果进行重排序
   - 返回最终的检索结果
4. **关键参数**：
   - 融合权重：`FUSION_WEIGHTS = {"structure": 1.3, "fts": 1.0, "vector": 0.9}`
   - 每个信号的候选结果数：结构检索 20 条，全文检索 80 条，向量检索 40 条
   - 最终返回结果数：8 条
5. **为什么这么设计**：
   - 多信号检索能提高初始检索的准确性
   - RRF 融合能有效结合不同检索方式的优势
   - 重排序能进一步提高结果的相关性
6. **替代方案**：
   - 仅使用向量检索：实现简单，但效果可能较差
   - 使用其他融合算法（如线性融合）：效果可能不如 RRF
   - 不进行重排序：简化流程，但结果质量可能下降
7. **trade-off**：
   - 检索准确性 vs 性能
   - 简单性 vs 复杂度
   - 召回率 vs 精确率

   - "为什么使用三种检索信号？"

   - "RRF 算法的原理是什么？"
   - "如何并行执行三种检索？"
9. **标准回答**：
   - "我们使用三种检索信号是因为它们覆盖了不同的检索维度：结构信号能利用文档的自然层级关系，全文信号对关键词匹配敏感，向量信号能捕捉语义相关性。这三种信号互补，结合起来能提高检索的全面性和准确性。"
   - "权重是基于经验和初步测试确定的，结构信号权重最高（1.3）是因为标题和章节信息通常最能反映内容主题，全文信号次之（1.0），向量信号最低（0.9）。后续可以基于用户反馈和 A/B 测试动态调整。"

   - "目前三种检索是串行执行的，但每个检索内部都有优化，比如向量检索使用了 Qdrant 的混合检索，全文检索使用了 PostgreSQL 的 FTS 索引。后续可以考虑将它们并行化，进一步降低延迟。"

5. **幻觉检测**：有 prompt 约束，但缺乏独立的检测机制

**面试措辞建议**：
- 使用"我们在设计中考虑了"、"目前实现了基础的"等保守性表述
- 明确说明当前实现的范围和限制
- 可以结合"未来计划进一步优化"等表述

**容易被问穿的地方**：
- 高并发的具体测试数据和性能指标
- 评估机制的具体实现和使用情况
- 多语言支持的具体测试和验证

**如何避免夸大**：
- 明确说明当前实现的阶段和范围
- 不使用"完全支持"、"已经实现"等绝对表述
- 强调"正在改进"、"计划扩展"

### C. 只能作为优化思路讲的

1. **基于语义的切片**：当前使用固定窗口切片
2. **先进的重排序模型**：当前使用简单的重排序
3. **动态参数调整**：当前使用固定参数
4. **全面的可观测性**：当前只有基础日志
5. **用户反馈集成**：当前未实现用户反馈收集
6. **独立的事实核查机制**：当前依赖 prompt 约束
7. **多模态支持**：当前只支持文本

**面试措辞建议**：
- 使用"如果重新设计，我会考虑"、"未来优化方向包括"等表述
- 明确说明这是"优化思路"而非"已实现功能"
- 可以结合当前系统的局限性展开

**容易被问穿的地方**：
- 为什么当前没有实现
- 实现这些功能的技术难点
- 实现这些功能的预期效果

**如何避免夸大**：
- 明确区分"已实现"和"优化思路"
- 不将优化思路说成"即将实现"
- 强调这些是"行业最佳实践"和"技术发展趋势"
